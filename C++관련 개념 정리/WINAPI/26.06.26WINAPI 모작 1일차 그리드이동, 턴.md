# 모작 제작 과정 — 그리드 이동 & 턴 시스템 구현

## 1. 기존 프레임워크 재사용 전략

기존 학습용 프레임워크(싱글톤 매니저 기반 Win32 GDI 구조)를 베이스로 사용하기로 결정. 새 프로젝트로 처음부터 다시 시작하는 대신, 기존 클래스(Player, Monster, Bullet, Shield 등)는 그대로 보존하고 신규 클래스를 추가하는 방식을 채택했다.

**이유**:
- 기존 클래스들은 서로 참조 관계가 복잡하게 얽혀 있어, 사용하지 않는 기능을 코드에서 직접 제거하면 의존성 연쇄로 컴파일 오류가 발생할 위험이 큼.
- 안 쓰는 기능을 "제거"하지 않고 "호출하지 않는" 방식으로 두면 이런 위험이 사라짐. 완성 후 안정성이 확인된 시점에 실제로 불필요한 클래스를 정리하기로 함 — 정리 작업의 리스크를 가장 낮은 시점으로 미루는 전략.

## 2. 그리드(칸 단위) 이동 시스템

기존 프레임워크는 캐릭터 위치를 픽셀 단위 실수(float)로 관리하는 실시간 이동 구조였다. 턴제 시스템에서는 "한 번의 입력 = 정확히 한 칸 이동"이 보장되어야 하므로, 위치를 row/col 정수 좌표로 관리하는 방식으로 전환했다.

- Struct.h에 GRIDPOS(row, col) 구조체 추가.
- 공통 부모 클래스 Obj에 m_tGridPos 멤버와 Set_GridPos() 함수를 추가. 칸 좌표를 갱신하면 픽셀 좌표(fX, fY)도 함께 자동 계산되도록 만들어, 기존 렌더링 코드를 수정하지 않고도 그리드 이동을 적용할 수 있게 했다.
- 신규 클래스 NewPlayer를 만들어 그리드 이동을 구현. 키 입력 판정을 Key_Pressing(누르는 동안 연속 처리) 대신 Key_Down(누른 순간 1회)으로 변경해, 한 번 입력에 한 칸만 이동하도록 했다.

### 핵심 코드 — 칸 좌표 변경 시 픽셀 좌표도 함께 갱신
```cpp
void Set_GridPos(int _row, int _col)
{
	m_tGridPos.row = _row;
	m_tGridPos.col = _col;

	// 기존 Render()는 좌상단 픽셀 좌표(fX, fY) 기준으로 그리므로,
	// 칸 좌표가 바뀔 때마다 자동으로 그 위치를 계산해둔다.
	m_tInfo.fX = (float)(_col * TILECX);
	m_tInfo.fY = (float)(_row * TILECY);
}
```

## 3. 타일 충돌(벽 판정) 처리

기존 TileMgr는 타일의 그림 번호(DrawID)만 관리하고, 통행 가능 여부를 판단하는 기능이 없었다. TileMgr에 Is_Wall(row, col) 함수를 추가해, 그림 번호가 특정 값(WALL_DRAWID) 이상이면 벽으로 판정하도록 구현했다.

칸 좌표를 1차원 배열 인덱스로 변환(row * TILEX + col)하는 과정에서 경계값 처리가 중요했다. 인덱스가 음수가 되는 경우(상단/좌측 경계 초과)와, 배열 크기를 초과하는 경우(하단/우측 경계 초과)를 모두 방어적으로 처리해 배열 범위 밖 접근으로 인한 크래시를 방지했다. 이동 가능 여부를 판단하는 NewPlayer::Can_Move()와 TileMgr::Is_Wall() 양쪽에서 동일한 경계 체크를 중복으로 두어, 어느 한쪽이 단독으로 호출되더라도 안전하게 동작하도록 설계했다.

부호 있는 정수(int)와 부호 없는 정수(size_t)를 함께 비교할 때는, 음수가 부호 없는 타입으로 변환되면 매우 큰 양수가 되는 점을 고려해 항상 부호 있는 타입 비교를 먼저 수행하는 순서를 지켰다.

### 핵심 코드 — 1차원 인덱스 변환과 경계 방어
```cpp
bool CTileMgr::Is_Wall(int _row, int _col)
{
	int iIdx = _row * TILEX + _col;

	// 음수(상/좌 경계 초과)는 0 > iIdx에서, size 초과(하/우 경계 초과)는
	// 뒤쪽 조건에서 각각 걸러진다. int로 먼저 비교해야 음수가 거대한
	// 양수로 잘못 변환되는 문제를 피할 수 있다.
	if (0 > iIdx || m_vecTile.size() <= (size_t)iIdx)
		return true;

	return m_vecTile[iIdx]->Get_DrawID() >= WALL_DRAWID;
}
```

## 4. 컬링 및 좌표 변환

TileMgr::Render()는 던전 전체(600개 타일)를 매 프레임 그리지 않고, 화면에 실제로 보이는 영역만 골라서 그리는 컬링을 적용한다. 스크롤 값(픽셀 단위)을 타일 크기로 나눠 칸 단위 인덱스로 변환하는 방식이며, 이는 "칸 번호 × 타일 크기 = 픽셀 좌표"라는 Set_GridPos()의 변환과 정반대 방향의 연산이다.

### 핵심 코드 — 픽셀 스크롤 값을 칸 단위 인덱스로 변환
```cpp
int iCullX = abs((int)CScrollMgr::Get_Instance()->Get_ScrollX() / TILECX);
int iCullEndX = iCullX + (WINCX / TILECX) + 2;   // 화면 경계에서 잘려 보이는 것을 방지하는 여유분
```

## 5. 화면 표시 방식 — 스크롤 + 시야(Fog of War)

원작은 던전 한 방이 화면보다 크고, 탐험한 영역만 시야로 밝혀지는 구조다. 이를 반영해 화면 고정 방식이 아닌 스크롤 방식을 그대로 유지하고, 시야 시스템(추후 VisionMgr로 구현 예정)을 필수 구현 항목에 포함시켰다. 시야는 단순한 시각 효과가 아니라 적의 위치를 가늠하는 전략 요소이므로 핵심 시스템으로 분류했다.

## 6. 턴 시스템 구조 결정 — 공식 자료 기반 재검토

초기에는 "플레이어 행동에 적이 즉시 반응하는" 단순한 구조를 가정했으나, 원작 공식 위키를 직접 조사한 결과 실제로는 "플레이어가 한 번 행동하면, 같은 맵의 모든 NPC가 순서대로 한 번씩 턴을 수행한 뒤에야 플레이어가 다시 턴을 쓸 수 있는" 페이즈 전환형 구조임을 확인하고 설계를 수정했다.

추가로 원작은 전투 중이 아닐 때 여러 턴을 자동으로 빠르게 처리해 실시간처럼 보이게 하는 별도 모드가 있다는 것도 확인했으나, 이는 구현 우선순위에서 후순위로 분류했다.

## 7. TurnMgr 구현

확인된 구조(페이즈 전환형)를 기준으로 구현.

- Obj에 On_Turn() 가상함수 추가(기본 구현은 빈 함수로 두어, 턴마다 행동이 필요 없는 오브젝트는 오버라이드하지 않아도 되게 함). Is_Dead() getter 추가.
- ObjMgr에 Get_MonsterList() 추가 — 살아있는 몬스터 전체를 순회할 수 있는 통로 제공.
- TurnMgr: PLAYER_TURN / ENEMY_TURN 두 상태를 가지며, 적 턴이 되면 몬스터 리스트를 순회해 살아있는 개체에 한해 On_Turn()을 호출하고 다시 플레이어 턴으로 복귀.
- NewPlayer는 플레이어 턴일 때만 입력을 받고, 실제로 이동에 성공했을 때만(벽에 막혀 이동이 무산된 경우는 제외) 턴을 종료시켜 적 턴으로 넘긴다. 막힌 방향키 입력으로 턴이 소모되지 않도록 한 설계다.
- 핵심 로직은 Windows API를 최소 더미로 대체한 독립 실행 코드로 컴파일·실행하여, 턴 전환과 몬스터 순회, 사망 개체 제외 동작을 직접 검증했다.

### 핵심 코드 — 적 턴 처리와 다시 플레이어 턴으로의 복귀
```cpp
void CTurnMgr::Process_EnemyTurn()
{
	const list<CObj*>& listMonster = CObjMgr::Get_Instance()->Get_MonsterList();

	for (auto& pMonster : listMonster)
	{
		if (pMonster->Is_Dead())
			continue;
		pMonster->On_Turn();
	}

	m_eTurnState = PLAYER_TURN;
}
```

## 8. 클래스 설계 원칙 — 개별 클래스가 필요한 기준

Tile/TileMgr, Obj/ObjMgr처럼 "개별 클래스 + 관리자" 구조를 일관되게 적용하는 한편, Turn처럼 동시에 하나만 존재하는 개념은 별도 클래스 없이 매니저 내부의 상태값(enum)으로 표현했다.

적용 기준: "자기만의 데이터와 동작(함수)을 가진 단위가 동시에 여러 개 존재해야 하는가". 타일이나 오브젝트는 이 조건을 만족하므로 개별 클래스가 필요하지만, 턴은 항상 하나의 상태만 존재하므로 변수 하나로 충분하다. 이후 출혈·디버프처럼 "여러 턴 동안 지속되는 효과"를 추가할 경우에도, 늘어나는 것은 턴을 세는 카운터 값이지 턴 객체 자체가 아니며, 실제로 여러 개로 관리해야 하는 대상은 각 오브젝트가 보유하는 디버프 목록이라는 결론을 정리했다. KeyMgr가 여러 키 상태를 배열 하나로 관리하는 것도 같은 원칙의 사례로 확인했다.

### 핵심 코드 — 상태값 하나로 충분한 경우
```cpp
enum TURN_STATE { PLAYER_TURN, ENEMY_TURN };

class CTurnMgr
{
	TURN_STATE m_eTurnState;   // 턴은 항상 하나만 존재하므로, 개별 클래스 대신 상태값으로 충분
};
```
