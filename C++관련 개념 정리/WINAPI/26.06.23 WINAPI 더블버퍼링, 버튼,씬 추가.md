# WINAPI 더블버퍼링, 버튼, 씬 추가

## 1. `lstrcmp`와 `!`(논리 NOT)

- `lstrcmp(a, b)`는 Win32 API의 문자열 비교 함수로, **두 문자열이 같으면 0을, 다르면 0이 아닌 값**을 반환함 (C의 `strcmp`와 동일한 관례)
- `strcmp`도 똑같이 "같으면 0"을 반환함 — C 계열 문자열 비교 함수들의 공통 설계
- `!lstrcmp(a, b)`처럼 `!`를 붙이는 이유:
  - C/C++에서 `true`는 1, `false`는 0으로 취급됨
  - `lstrcmp`는 "같으면 0(false처럼 보임)"이라 직관과 반대로 동작하므로, `!`로 뒤집어서 "같으면 true(1)"가 되도록 맞춰주는 것
  - `!`는 값을 "1로 바꾸는 연산자"가 아니라 "참/거짓을 뒤집는 논리 연산자"이며, 0이 아닌 모든 값은 참으로 취급되어 뒤집히면 거짓(0)이 됨

## 2. `const TCHAR*`에서 `const`를 붙이는 이유

- `CStrCmp(const TCHAR* _pString)`처럼 `const`를 붙인 이유:
  - 이 클래스는 문자열을 비교만 하고 수정할 일이 없음
  - "이 값은 절대 안 바뀐다"는 의도를 컴파일러에 보증시켜 실수 방지
  - **문자열 리터럴**(`L"hello"` 등)은 본질적으로 const 데이터로 취급되어, `const`가 없는 포인터로는 받을 수 없음 (최신 C++ 컴파일러에서는 컴파일 에러)
- 단, "일반 변수(배열, 버퍼 등)"를 넘길 때는 `const` 여부와 무관하게 둘 다 가능 — 문제가 되는 건 **리터럴을 넘길 때만**
- 결론: 수정할 필요가 없는 포인터는 **const를 붙이는 것이 거의 항상 더 유리함** (받을 수 있는 입력 범위가 넓어지고, 의도도 명확해짐) — 이를 "const correctness(상수 정확성)"라고 부름

## 3. `auto` vs `auto&` — find_if를 중심으로

- `auto iter = find_if(...)`에서 `auto&`가 아니라 `auto`를 쓴 이유에 대한 탐구
- **핵심 이유**: `find_if`의 반환값은 함수가 끝나며 즉석에서 만들어지는 **임시 값(rvalue)**이라서, 일반 참조(`auto&`, non-const lvalue reference)로는 **문법적으로 받을 수 없음** (컴파일 에러)
  - 비유: `int& y = Get_Five();`처럼, 이름 없는 임시 결과는 참조로 못 붙잡음
- 부수적 이유: 이터레이터 자체가 가벼운 타입이라 복사 비용이 거의 없음 — 다만 이는 보조적인 이유일 뿐, **rvalue라는 문법적 제약이 결정적인 이유**
- 일반적인 경우(이미 존재하는 컨테이너를 `for`로 순회할 때, 예: `for (auto& pBmp : m_mapBmp)`)는 **lvalue**이므로 `auto&`가 정상적으로 가능하고, 이런 경우엔 가벼운 타입이라도 `auto&`(혹은 `const auto&`)를 쓰는 것이 일반적으로 더 권장됨
- 정리: "find_if에서 auto를 쓴 이유는 가벼워서가 아니라, find_if의 반환값이 rvalue라는 특수성 때문"이라는 결론으로 합의됨

## 4. lvalue / rvalue 개념

- **lvalue**: 이름이 있고 메모리상 실제 주소를 가져서 계속 참조 가능한 대상 (`&`로 주소를 구할 수 있음). 예: 변수, 이미 존재하는 컨테이너(`m_mapBmp`)
- **rvalue**: 이름이 없고 그 순간 계산되어 나타났다가 사라지는 임시 값. 예: 리터럴(`5`), 산술 연산 결과(`x+3`), `find_if(...)`의 반환값
- 이름의 유래: 대입문 `x = 5`에서 좌변(left)에 올 수 있는 것이 lvalue, 우변(right)에 오는 임시 결과가 rvalue
- rvalue는 일반 참조(`&`)로 묶을 수 없다는 규칙이 `auto&` 관련 논의의 핵심 근거였음

## 5. `find_if` 함수 자체에 대한 정리

- `<algorithm>` 헤더의 함수로, **begin부터 end까지 순회하며 조건자(predicate)가 true를 반환하는 첫 원소의 위치(이터레이터)를 반환**, 못 찾으면 `end()` 반환
- 조건자로 일반 함수, 함수 객체(Functor, 예: `CStrCmp`), 람다 중 하나를 사용 가능
- `find_if`는 **map 전용이 아니라 vector, list, set, map 등 이터레이터가 있는 모든 컨테이너에서 범용으로 사용 가능**한 알고리즘
- map과 자주 같이 보였던 이유: map은 원래 `map.find(key)`로 매우 빠른(O(log n)) 검색이 가능하지만, 키 타입이 포인터(`const TCHAR*`)인 경우 주소값만 비교되어 문자열 내용 비교가 안 되므로, 이를 우회하기 위해 `find_if` + `CStrCmp`(직접 `lstrcmp`로 내용 비교)를 사용한 특수한 사례였음

## 6. vector/list/map의 "탐색" 비교 (자료구조 복습 및 보완)

- 기존 학습 내용(정확함): vector는 임의 접근(인덱스 접근, `v[i]`)이 빠르지만 중간 삽입/삭제가 불리(O(n), 뒤를 밀어야 함); list는 임의 접근이 불리하지만 삽입/삭제가 유리(링크만 끊고 이으면 됨, O(1)); map은 양쪽의 중간적 성격(삽입/삭제/검색 모두 O(log n), 트리 구조)
- 추가로 구분된 내용: "탐색"이라는 단어가 두 가지 다른 의미로 쓰임
  - **인덱스 접근(임의 접근)**: vector 유리, list 불리 — 기존에 배운 내용
  - **조건 검색(`find_if`처럼 "조건에 맞는 원소 찾기")**: vector와 list **둘 다 동일하게 O(n)**, 이론적으로는 우열 없음
- 실전 성능 차이: 이론적 시간복잡도(Big-O)는 vector와 list가 동일(O(n))하지만, **실제 실행 속도는 캐시 지역성(Cache Locality) 때문에 vector가 보통 더 빠름** — vector는 메모리가 연속적으로 배치되어 캐시 적중률이 높고, list는 메모리가 흩어져 있어 포인터를 따라가며 캐시 미스가 발생하기 쉬움

## 7. `CBmpMgr::Insert_Bmp` / `Find_Image` / `Release` 코드 해석

- `Insert_Bmp`: `find_if`로 같은 키(`_pImageKey`)가 이미 등록되어 있는지 확인하고, 없으면(`end()`와 같으면) 새로 `CMyBitmap`을 만들어 `Load_Bmp`로 로드한 뒤 `map.emplace`로 등록 — **중복 로딩 방지** 목적
- `Find_Image`: 같은 방식으로 검색 후, 못 찾으면 `nullptr` 반환(반환 타입이 `HDC`라 "없음"을 표현하는 가장 자연스러운 방법), 찾으면 `iter->second->Get_DC()`로 해당 비트맵의 HDC를 반환
  - `iter->second`는 **map의 pair 중 value 값**(`first`=key, `second`=value)을 가져오는 것이며, `CMyBitmap*` 타입이므로 `->Get_DC()`로 그 객체의 HDC를 꺼내옴
- `Release`: `for (auto& pBmp : m_mapBmp)`로 모든 원소를 순회하며 `SAFE_DELETE(pBmp.second)`로 각 value(CMyBitmap 객체)를 메모리에서 해제하고, `map.clear()`로 컨테이너 자체도 비움
  - 이때 `auto&`가 가능한 이유는 "map이라서"가 아니라, **이미 존재하는 컨테이너(`m_mapBmp`, lvalue)를 직접 순회하기 때문** — map만의 특별한 성질이 아니라 vector, list 등 어떤 컨테이너든 동일하게 적용되는 원리

## 8. 유니코드와 `L"문자열"` / `_T(...)`

- `L"Start"`의 `L`은 **유니코드(wide character, wchar_t 기반) 문자열 리터럴**을 의미하는 접두사
- `TCHAR`는 프로젝트의 문자 집합 설정에 따라 `char`(MBCS) 또는 `wchar_t`(유니코드)로 자동 전환되는 타입이며, 지금 학습 코드는 유니코드 설정이므로 `TCHAR`=`wchar_t`이고, 이와 타입을 맞추기 위해 `L"문자열"`을 사용해야 함
- **유니코드란**: 전 세계 모든 언어의 모든 문자에 겹치지 않는 고유 번호를 부여한 국제 표준 체계. 과거에는 나라별로 다른 인코딩(한국 CP949, 일본 Shift-JIS 등)을 썼기 때문에 호환성 문제가 잦았고, 유니코드로 통일하면 이 문제가 해결됨
- **`_T(...)` / `TEXT(...)` 매크로**: 프로젝트의 문자 집합 설정에 따라 자동으로 `"문자열"`(MBCS) 또는 `L"문자열"`(유니코드)로 치환해주는 매크로. `auto`가 타입을 자동으로 추론해주는 것과 비슷한 발상으로, **TCHAR를 쓰는 프로젝트라면 일관성을 위해 `L"..."`보다 `_T("...")`를 쓰는 것이 더 권장됨**
  - 단, `_T`는 함수가 아니라 매크로이므로 **반드시 괄호로 감싸서** `_T("문자열")` 형태로 써야 하며, `_T"문자열"`처럼 괄호 없이 쓰면 컴파일 에러(인수 부족 등)가 발생함
- 게임업계가 유니코드를 표준으로 쓰는 이유: 전 세계 동시 출시(글로벌 런칭)가 흔한 산업 특성상, 여러 언어를 한 체계로 통일해서 다뤄야 호환성 문제가 없기 때문 (채팅, UI, 아이템 이름 등 다국어 처리에 필수적)

## 9. `PtInRect` 함수와 버튼 클릭/호버 판정

- `PtInRect(&rc, pt)`: 점(`pt`)이 사각형(`rc`) 영역 안에 있는지 검사하는 Win32 함수. 있으면 TRUE, 없으면 FALSE 반환
  - 경계 규칙: left/top은 포함, right/bottom은 미포함 (이전에 본 `LineMgr::Collision_Line`의 끝점 비교 관례와 유사한 설계)
- `CMyButton::Late_Update()`에서의 활용:
  - `GetCursorPos` + `ScreenToClient`로 마우스의 창 내부 좌표를 구하고
  - `PtInRect(&m_tRect, pt)`로 **마우스 좌표가 버튼 이미지의 사각형 영역과 겹치는지(호버 상태)**를 판정
  - 겹친 상태에서 `Key_Down(VK_LBUTTON)`이면 버튼별(`Start`/`Edit`/`Exit`) 동작 실행, 그 외엔 `m_iDrawID`를 통해 호버/기본 이미지를 전환
- `Render()`의 `GdiTransparentBlt`에서 `m_iDrawID * 150`은 하나의 이미지 파일 안에 기본/호버 이미지가 가로로 나란히 배치되어 있고, 그중 어느 영역(150px 단위)을 잘라 보여줄지 선택하는 것

## 10. `VK_RETURN`과 가상 키 코드(Virtual Key Code)

- `VK_RETURN`은 16진수 `0x0D`(Enter 키)를 가리키는 이름 있는 상수로, `VK_`(Virtual Key) 접두사 + 키 이름으로 구성된 Win32의 명명 체계
- 직접 숫자(`0x0D`)를 쓰는 대신 `VK_RETURN`처럼 이름을 쓰는 이유는 가독성 (어떤 키인지 코드만 보고 바로 알 수 있음)
- **`VK_ENTER`는 존재하지 않음** — `VK_RETURN`이 곧 Enter 키를 가리키는 유일한 상수이며, 이름이 "RETURN"인 이유는 타자기 시절 캐리지를 되돌리는 동작에서 유래한 역사적 명칭이 그대로 굳어진 것
- (참고) Numpad의 Enter는 같은 코드값을 공유하되 확장 키 플래그로 구분되는 저수준 처리가 있으나, 일반적인 입력 처리에서는 `VK_RETURN` 하나로 충분함

## 11. `BitBlt` 함수와 `GdiTransparentBlt`와의 비교

- `BitBlt(hdcDest, x, y, cx, cy, hdcSrc, x1, y1, rop)`: 원본 이미지(비트맵)의 특정 영역을 대상 DC의 특정 위치에 그대로 복사하는 GDI 함수 ("Bit Block Transfer"의 줄임말)
- `BitBlt(_DC, 0, 0, WINCX, WINCY, hLogoDC, 0, 0, SRCCOPY)`의 의미: 로고 이미지의 (0,0)부터 800x600 영역을 화면(_DC)의 (0,0) 위치에 투명 처리 없이 그대로 복사
- `SRCCOPY`는 가장 기본적인 raster operation으로, 원본을 그대로 덮어써서 복사하는 방식
- `BitBlt` vs `GdiTransparentBlt` 비교:
  - `BitBlt`: 투명 처리 없음 — 배경, 로고, 메뉴 배경처럼 화면 전체를 꽉 채우는 불투명 이미지에 사용
  - `GdiTransparentBlt`: 특정 색(흰색 등)을 투명 처리 — 캐릭터, 버튼처럼 다른 요소와 겹쳐 보여야 하는 이미지에 사용

## 12. 더블 버퍼링(Double Buffering) — BackBuffer의 역할

- `BackBuffer` 이미지 자체가 화면에 직접 보이지 않는 이유에 대한 질문에서 시작
- `MainGame::Render()`의 구조:
  ```cpp
  HDC hBackDC = Find_Image(_T("Back"));               // 중간 작업대
  HDC hBBackBufferDC = Find_Image(_T("BackBuffer"));   // 깨끗한 빈 배경

  BitBlt(hBackDC, 0, 0, WINCX, WINCY, hBBackBufferDC, 0, 0, SRCCOPY);
  // ① BackBuffer(빈 배경)를 Back(중간 작업대)에 복사해서, 이전 프레임 잔상을 지움

  CSceneMgr::Get_Instance()->Render(hBackDC);
  // ② 그 위에 현재 씬(로고/메뉴/게임)을 그림

  BitBlt(m_hDC, 0, 0, WINCX, WINCY, hBackDC, 0, 0, SRCCOPY);
  // ③ 완성된 결과(hBackDC)를 진짜 화면(m_hDC)에 최종적으로 한 번에 복사
  ```
- **더블 버퍼링의 핵심 개념**: 화면에 직접 그리지 않고, 보이지 않는 별도의 메모리 공간(백버퍼)에 먼저 모든 그림(지우기+그리기)을 완성해놓은 뒤, 완성된 결과를 화면에 한 번에 옮겨서 **깜빡임(flickering)을 방지**하는 기법
  - `BackBuffer`(빈 배경 이미지)는 매 프레임마다 작업대(`hBackDC`)를 깨끗하게 리셋하는 용도일 뿐, 화면에 직접 보여지는 대상이 아님 — 항상 `hBackDC`라는 중간 단계를 거쳐서만 화면에 도달하므로, BackBuffer 원본이 단독으로 화면에 보이지 않는 것은 정상
- **다른 학습 단계(05_ObjMgr_총알/추적AI)와의 비교**: 그 단계에서는 `Rectangle(m_hDC, 0, 0, WINCX, WINCY)`로 **화면(m_hDC)에 직접** 지우고 다시 그리는 가장 단순한 방식을 사용 — "잔상을 지운다"는 목적은 BackBuffer 방식과 동일하지만, 방법(직접 화면을 지우는 것 vs 메모리에서 미리 그려서 한 번에 옮기는 것)은 다름
  - 비유: `Rectangle()` 방식은 칠판을 직접 지우개로 지우고 다시 그리는 것(지우는 순간이 보여 깜빡임 가능), BackBuffer 방식은 연습장에 미리 다 그려놓고 완성되면 칠판에 한 번에 옮겨붙이는 것(지우는 과정이 안 보여 깜빡임 없음)
  - 학습 흐름상, 단순한 `Rectangle()` 지우기 방식에서 이후 더블 버퍼링(`BackBuffer`) 방식으로 발전된 것으로 추정됨
