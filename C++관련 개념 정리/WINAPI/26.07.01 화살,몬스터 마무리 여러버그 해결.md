# 모작 제작 과정 — 전투 시스템 디버깅 및 안정화

## 1. 체력/데미지 시스템 설계

체력(HP)과 공격력(데미지)을 어디에 둘지 논의했다. 체력은 모든 캐릭터가 공통으로 가지는 상태이므로 최상위 부모 클래스에 두고, 공격력은 이미 공격 방식 자체가 클래스별로 나뉘어 있어 각 공격 주체 클래스에 따로 두는 것으로 정리했다. "데미지를 받는 함수까지 공통 부모에 두면, 화살처럼 공격받을 일이 없는 오브젝트도 문법적으로 데미지를 받을 수 있게 되는 것 아닌가"라는 의문이 있었는데, 이는 다형성을 위한 공통 인터페이스로서 필요한 설계이며, 실제로 그렇게 호출하는 코드가 없는 한 위험은 없다는 결론으로 정리했다.

### 핵심 코드 — 공통 체력/데미지 처리
```cpp
// Obj.h
void Take_Damage(int _dmg)
{
	m_iHp -= _dmg;
	if (m_iHp <= 0)
		m_bDead = true;
}
```

## 2. 몬스터 공격력 구현

근접/원거리 몬스터 각각에 공격력 멤버를 추가하고, Attack() 함수 안에서 대상의 Take_Damage()를 호출하도록 구현했다. 공격-데미지 누적-사망까지 이어지는 전체 흐름을 직접 컴파일한 코드로 검증해, 의도한 턴 수에 정확히 사망 판정이 이루어짐을 확인했다.

### 핵심 코드 — 공격 시 데미지 적용
```cpp
void CMonster_Melee::Attack(CObj* _pTarget)
{
	_pTarget->Take_Damage(Get_Atk());
}
```

## 3. 턴 진행 중 플레이어 사망 시 안전장치 부재 발견

몬스터의 공격으로 플레이어가 죽는 경우, 턴 관리자가 이를 확인하지 않고 계속 턴을 진행시키는 문제를 발견했다. 플레이어가 죽어 오브젝트 목록에서 삭제된 이후에도 몬스터의 행동 로직이 플레이어를 찾으려 시도하면, 빈 목록에서 첫 원소를 꺼내려는 정의되지 않은 동작으로 이어질 수 있는 잠재적 위험이었다.

세 단계의 방어 코드를 추가했다. 턴 관리자에서 적 턴이 끝난 직후 플레이어 생존 여부를 확인해 게임 진행을 멈추도록 했고, 오브젝트 매니저의 플레이어 조회 함수 자체도 목록이 비어 있으면 안전하게 빈 값을 반환하도록 수정했으며, 몬스터의 행동 함수에도 빈 값을 받았을 때 즉시 종료하는 방어 코드를 추가해 이중삼중으로 안전하게 만들었다.

### 핵심 코드 — 적 턴 종료 시 플레이어 생존 확인
```cpp
void CTurnMgr::Process_EnemyTurn()
{
	const list<CObj*>& listMonster = CObjMgr::Get_Instance()->Get_MonsterList();
	for (auto& pMonster : listMonster)
	{
		if (pMonster->Is_Dead())
			continue;
		pMonster->On_Turn();
	}

	CObj* pPlayer = CObjMgr::Get_Instance()->Get_Player();
	if (pPlayer && pPlayer->Is_Dead())
	{
		m_bGameOver = true;
		return;
	}
	m_eTurnState = PLAYER_TURN;
}
```

### 핵심 코드 — 빈 목록 안전 조회
```cpp
CObj* Get_Player() { return m_listObj[OBJID::PLAYER].empty() ? nullptr : m_listObj[OBJID::PLAYER].front(); }
```

이 수정을 적용하기 전 상태로 실제 테스트했을 때는 우연히 크래시가 발생하지 않았는데, 이는 버그가 없었다는 의미가 아니라 특정 실행 순서에서 우연히 문제가 드러나지 않은 것일 뿐이라는 점을 확인했다. 정의되지 않은 동작은 실행 환경이나 타이밍에 따라 결과가 달라질 수 있으므로, 한 번 안전하게 동작했다고 해서 안전성이 보장되는 것은 아니라는 점을 짚었다.

## 4. 화면이 전부 검게 표시되는 문제의 오진단과 재진단

전투 시스템 작업 직후, 인트로 영상 재생 이후 화면이 흰색으로 멈추는 현상이 발생했다. 처음에는 직전에 수정한 턴 관리 로직이 원인이라 추정하고 그 방향으로 점검했으나, 실제로는 게임을 플레이해보지도 못한 채(메뉴 화면조차 뜨지 않은 채) 발생한 현상이라는 것을 사용자가 정확히 지적해, 추정이 잘못되었음을 확인했다.

메뉴 화면을 그리는 코드를 추적한 결과, 메뉴 배경과 버튼에 쓰이는 이미지 파일 자체가 작업 폴더에 없다는 것을 발견했다. 저작권 문제로 스프라이트 이미지를 별도 보관하고 코드만 압축해 전달하는 과정에서, 이미지 폴더를 다시 포함시키지 않았던 것이 원인이었다. 코드 결함이 아니라 전달 과정의 누락이었음을 확인하고, 앞서 발견했던 턴 관리 관련 안전장치는 별개로 유효한 수정이므로 그대로 적용하기로 했다.

## 5. 캐릭터/투사체 렌더링 좌표 및 크기 불일치 (반복 패턴)

플레이어와 몬스터가 화면에 보이지 않는 문제를 디버그 텍스트로 좌표와 이미지 핸들을 직접 출력해가며 단계적으로 좁혔다. 원인은 두 가지였다. 첫째, 원본 스프라이트의 실제 픽셀 크기와 그리기 함수에 넘긴 크기(그리드 한 칸 크기)가 달라, 존재하지 않는 영역을 읽으려는 시도가 발생하고 있었다. 둘째, 화면에 그릴 때 스크롤 값을 좌표에 더하는 보정이 빠져 있어, 캐릭터가 스크롤 진행에 따라 점점 어긋난 위치에 그려지고 있었다.

이 두 가지 문제는 이후 화살 투사체 클래스에서도 동일하게 재발했다. 화살 전용 클래스가 가장 나중에 추가된 클래스였기 때문에, 앞서 다른 클래스에서 처리했던 두 가지 보정(원본 크기 일치, 스크롤 보정)이 처음부터 누락된 채로 작성되어 있었던 것이다. 같은 원인이 반복되는 것을 보고, 새로운 렌더링 가능 클래스를 추가할 때는 이 두 가지를 기본 점검 항목으로 삼아야 한다는 점을 정리했다.

### 핵심 코드 — 원본 크기와 스크롤을 함께 반영한 렌더링
```cpp
int iScrollX = (int)CScrollMgr::Get_Instance()->Get_ScrollX();
int iScrollY = (int)CScrollMgr::Get_Instance()->Get_ScrollY();

GdiTransparentBlt(_DC,
	m_tRect.left + iScrollX, m_tRect.top + iScrollY,
	21, 21,
	hArrowDC,
	0, 0,
	21, 21,
	RGB(255, 255, 255));
```

## 6. 턴 진행 자체가 멈추는 문제

캐릭터가 보이게 된 이후, 한 번 이동한 뒤로 키 입력이 반응하지 않는 문제를 발견했다. 화면에 현재 턴 상태를 직접 출력해 확인한 결과 적 턴 상태에서 멈춰 있었고, 원인은 턴 관리자의 매 프레임 갱신 함수 자체가 스테이지의 갱신 루틴에서 호출되지 않고 있었던 누락이었다. 플레이어가 턴을 종료시키는 호출은 정상적으로 일어났지만, 그 상태를 받아 적 턴을 처리하고 다시 플레이어 턴으로 되돌리는 절차가 실행될 기회 자체를 얻지 못하고 있었다.

## 7. 입력 처리 — 키를 누르고 있는 동안 반복 이동되는 문제

턴 진행이 정상화된 뒤, 이동 속도가 과도하게 빠르게 느껴지는 문제를 확인했다. 대각선 이동을 지원하기 위해 매 프레임 키 상태를 확인하는 방식을 쓰고 있었는데, 그 결과로 한 번의 입력에 여러 프레임에 걸쳐 연속 이동이 발생하고 있었다. 별도의 입력 잠금 상태를 두어, 한 번 이동에 성공하면 즉시 잠그고 모든 방향키가 떨어지는 순간에만 잠금을 해제하는 방식으로 해결했다. 이를 통해 대각선 입력 판정 방식은 유지하면서도, 한 번의 입력에 정확히 한 칸만 이동하는 턴제 게임의 기본 동작을 보장했다.

## 8. 투사체 생명주기 길이 조정

화살 투사체가 한 프레임만 표시되도록 만들었던 기존 설정이 너무 짧아 거의 인지되지 않는 문제를 발견했다. 생명주기를 늘려 테스트하는 과정에서, 지나치게 길게 설정했을 때는 오히려 정지된 투사체가 마치 이동하는 것처럼 보이는 착시가 발생한다는 것도 함께 확인했다. 여러 차례 값을 조정하며 직접 테스트한 결과, 짧게 인지 가능하면서도 착시가 발생하지 않는 적정값을 찾아 적용했다.

## 9. 코드와 무관한 환경 문제 — 인코딩과 자산 경로

작업 과정에서 두 가지 환경적 문제를 확인하고 대응 방식을 정했다. 하나는 원본 파일이 한국어 윈도우 표준 인코딩으로 저장되어 있는데, 작업 도구가 보여주는 미리보기가 다른 인코딩을 기본으로 가정해 한글이 깨져 보이는 문제로, 이후 미리보기와 전달 파일 모두 호환되는 인코딩으로 통일해 해결했다. 다른 하나는 일부 투사체 이미지 파일의 경로가 실제 폴더 구조와 일치하지 않아 이미지를 찾지 못하던 문제로, 경로를 직접 점검해 해결했다.
