## 2. 발견한 버그/이슈 목록 (SoundMgr 원본 레거시 코드)

1. `map<TCHAR*, FMOD_SOUND*>` — 키가 포인터(주소값)라서 `map`의 정렬/탐색 장점을 못 살림. `wstring`으로 바꾸면 해결.
2. `MultiByteToWideChar` 버퍼 크기 계산 — `strlen()`(바이트 수)을 와이드 문자 버퍼 크기로 그대로 씀.
3. `_findfirst` 실패 체크가 `== 0`인데 실제 실패값은 `-1`.
4. `Release()`에서 `FMOD_System_Release`/`FMOD_System_Close` 순서가 거꾸로 (Close 먼저, Release 나중이 맞음).
5. `PlaySound`에서 `FMOD_Channel_IsPlaying`의 리턴값(`FMOD_RESULT`, 호출 성공 여부)을 재생 여부로 잘못 해석 — 실제 재생 여부는 `bPlay` 매개변수로 따로 받아야 함.
6. `m_pChannelArr`가 생성자에서 초기화되지 않음.
7. (추가 확인) `MyButton::Update()`가 일부 경로에서 리턴값 없음, `CGuideBullet`의 0으로 나누기 가능성.

## 6. `const TCHAR*` 컴파일 에러

`PlaySound(_T("tdryes00.wav"), ...)` 호출 시 `E0167` 에러 — 문자열 리터럴(`const wchar_t*`)과 함수 매개변수(`TCHAR*`, non-const) 타입 불일치.

- C++11부터 있던 표준 규칙이며, 최신 MSVC가 더 엄격히 적용해서 드러난 것 (VS 버전이 새로 만든 규칙이 아님).
- 해결책: `PlaySound`/`PlayBGM` 매개변수를 `const TCHAR*`로 변경.
- `m_mapSound`의 키 타입에 `const`를 붙이는 건 컴파일은 되지만, `delete[]`로 직접 해제할 책임이 있는 메모리에 "읽기 전용" 의미의 `const`를 붙이는 게 의미상 어색함 — 근본적 해결책은 `wstring` 전환이나, 학습 목적상 현재는 유지.

## 7. 코드 세부 Q&A (SoundMgr)

- **key/value 해제 방식이 다른 이유**: key(`TCHAR*`)는 `new[]`로 직접 만든 메모리라 `delete[]`로, value(`FMOD_SOUND*`)는 FMOD가 만든 객체라 `FMOD_Sound_Release`로 — "만든 주체가 해제해야 한다" 원칙.
- **`wstring`이 `map::find`를 가능하게 하는 원리**: `wstring`은 `operator<`가 문자열 내용(사전순) 비교로 정의되어 있어서 `map`(레드-블랙 트리)이 올바르게 정렬·탐색 가능. `TCHAR*`는 주소값 비교라 트리가 의미 없는 순서가 됨.
- **람다 문법**: `find_if(begin, end, [&](auto& iter){...})`에서 람다 전체가 세 번째 인자로 들어가는 구조. `[&]`(캡처), `(auto& iter)`(매개변수), `{...}`(본문)로 분해 설명.
- **`find_if` vs `find`**: 키 비교 상황에서는 `map::find`(O(log n))를 써야 하는데 키가 포인터라 못 쓰고 `find_if`(O(n))로 흉내내던 것 — 비효율. 단, 값(value)이나 다른 조건으로 찾을 때는 `find_if`가 유일한 선택지이므로 일반화는 금지.
- **`FMOD_BOOL`**: `typedef int FMOD_BOOL;` — 실제로는 `int`와 동일. C 기반 크로스플랫폼 라이브러리라 자체 타입을 둔 것.
- **`m_pChannelArr[MAXCHANNEL]`**: `MAXCHANNEL`이 enum의 6번째 항목(값 5)이라 배열 크기가 5가 되는 게 맞고, 이는 버그가 아니라 "enum 마지막 항목을 개수 카운터로 쓰는" 흔한 설계 패턴.

## 8. Monster.cpp — 회전 좌표 계산

```cpp
m_tPos[0].x = (LONG)(fCenterX + cosf((135.f + m_fAngle) * PI / 180.f) * fDis);
```

- 135°, 45°, 225°는 회전 없는 직사각형의 좌상단/우상단/좌하단이 중심으로부터 위치한 방향각(45°의 배수).
- "몬스터가 회전하는 게 아니라 사각형 틀(꼭짓점 좌표)이 회전하고, 그 틀에 비트맵이 끼워 맞춰지는 것"이라는 요약이 정확함을 확인.

## 9. `PlgBlt` 함수

- Parallelogram Blit. 목적지 평행사변형(점 3개)에 원본 사각형 영역을 매핑해서 그리는 GDI 함수.
- `BitBlt`(단순 복사) → `StretchBlt`(크기조절/반전) → `PlgBlt`(회전/기울임까지) 순으로 일반화.
- 평행사변형은 점 3개로 네 번째 점이 자동 결정되므로 점 3개만 필요.

## 10. Monster::Render의 3중 비트맵(Monster/Plg/Reset) 합성 트릭

업로드된 `Monster.bmp`(흰 배경 + 캐릭터)를 직접 확인함.

세 줄의 역할:

1. `PlgBlt(hPlgDC, ...)` — 회전된 몬스터를 "임시 작업용 캔버스(Plg)"에 그림. 평행사변형 바깥 네 귀퉁이는 처리 안 됨.
2. `GdiTransparentBlt(_DC, ..., RGB(255,255,255))` — 흰색을 투명 키로 화면(`_DC`)에 합성. 원본 배경(흰색)과 PlgBlt 처리 못한 귀퉁이(Plg가 흰색)가 동시에 투명 처리됨.
3. `BitBlt(hPlgDC, ..., hResetDC, ..., SRCCOPY)` — 다음 프레임을 위해 Plg를 다시 깨끗한 흰색(Reset)으로 덮어씀.

**Plg와 Reset이 둘 다 필요한 이유**: `Plg`는 매 프레임 그려지고 합성되고 리셋되는 "더러워지는 작업대", `Reset`은 절대 안 바뀌는 "깨끗한 원본 백업"으로, 역할이 다른 두 DC가 필요한 것이며 색이 같은(흰색) 것은 부수적 결과.

**흰색을 선택한 이유**: `Monster.bmp`의 배경색이 흰색이므로, 작업 캔버스(Plg/Reset)도 흰색으로 맞춰야 `GdiTransparentBlt` 한 번의 호출로 "원본 배경"과 "PlgBlt 처리 못한 귀퉁이"를 동시에 투명 처리할 수 있음.

## 11. 알파 채널 vs 알파 블렌딩

- 알파 채널: 픽셀에 저장된 투명도 "데이터"(RGBA의 A값).
- 알파 블렌딩: 그 데이터를 이용해 위 이미지와 배경을 섞는 "연산"(공식: `최종색 = 위색×알파 + 배경색×(1-알파)`).
- 현재 코드(GDI, 키 컬러 방식)는 알파 채널이 없던 시절 "흰색=투명"이라는 약속으로 투명도를 흉내 낸 것이고, 최신 API는 픽셀 자체에 진짜 투명도 정보를 담아 반투명 표현까지 가능하다는 차이.

## 12. Logo.cpp — 동영상 종료 판별 조건

```cpp
if (CKeyMgr::Get_Instance()->Key_Down(VK_RETURN)
    || MCIWndGetPosition(m_hVideo) >= MCIWndGetLength(m_hVideo))
```

- `MCIWndGetPosition`(현재 위치)과 `MCIWndGetLength`(전체 길이)를 비교해 동영상이 끝까지 재생됐는지 판별.
- `==`가 아니라 `>=`를 쓰는 이유(타이밍 오차 방어)도 확인.
- 엔터 스킵과 자연 종료, 두 진입 경로를 OR로 묶은 조건임을 확인.

## 13. MainGame — Back/BackBuffer BMP와 LOGO 분기의 관계

- `Back`/`BackBuffer`는 더블 버퍼링(작업 캔버스 + 깨끗한 원본 배경) 용도로, 로고 이후의 게임 화면을 그릴 때 쓰는 것.
- 로고 씬에서 `MCIWnd` 동영상이 화면에 떠 있는데, `Render()`가 매 프레임 `Back`을 화면 전체에 `BitBlt`로 덮어쓰면 동영상이 가려짐.
- LOGO 씬일 때 `Render()`를 즉시 종료시키는 코드와 BMP 사전 로드 코드는 서로의 부작용을 상쇄하는 짝. Monster의 Plg/Reset 패턴과 같은 원리.

## 14. ObjMgr::Render — OBJID 기준에서 RENDERID 기준으로 바뀐 이유

- `OBJID`(종류) 기준 분류는 화면상 앞뒤 가림 관계를 표현하지 못함 — 그리는 순서가 종류에 따라 고정되어 실제 위치와 무관하게 항상 같은 게 가려지는 문제.
- `RENDERID`(레이어: BACKGROUND → OBJECT → EFFECT → UI) 기준으로 큰 순서를 정하고, OBJECT 레이어 안에서는 Y좌표로 다시 정렬하는 2단계 구조로 바뀌었음.
- 메인의 더블 버퍼링 트릭과는 별개의, "그리는 순서(레이어링)" 문제 해결.

## 15. Y좌표 정렬의 원리

- 탑뷰/쿼터뷰 시점에서 화면 Y좌표가 클수록(아래쪽) 카메라와 가깝고, 작을수록(위쪽) 멀다는 약속.
- 가까운 오브젝트가 먼 오브젝트를 가려야 자연스러우므로, Y가 작은 것(먼 것)을 먼저 그리고 큰 것(가까운 것)을 나중에 그려서 덮어써야 함.
- X좌표로 정렬하면 위/아래 가림 관계는 표현되지 않고 좌우 그리는 순서만 결정된다는 점도 확인.

## 16. `CompareY<CObj*>` 템플릿과 `sort` 동작 원리

- `CompareY<CObj*>`는 함수를 호출하는 게 아니라 템플릿 타입 `T`를 `CObj*`로 고정하는 것이며, 실제 매개변수는 여전히 두 개(`_Dst`, `_Src`).
- `list::sort`가 리스트를 순회하며 원소를 두 개씩 꺼내 비교 함수에 넘기는 방식으로 동작.
- `m_listObj`(OBJID) → `m_listRender`(RENDERID)로 포인터가 재배치되는 과정에서, 자식 클래스가 부모 타입 `CObj*`로 업캐스팅되어 담기며 공통 인터페이스(`Get_Info()`)로 비교 가능.
- 직접 코드에 작성한 주석(템플릿/업캐스팅/Y정렬 이유/X정렬과의 차이/OBJID→RENDERID 전환 이유)이 전부 정확함을 확인.

## 17. Monster 회전 렌더링 버그 — 흰 사각형이 안 사라지는 문제

실행 화면 스크린샷에서 몬스터 주변 흰 사각형이 투명 처리되지 않고 그대로 보이는 문제. 두 zip(원본 학습자료, 직접 작성한 프로젝트)을 모두 압축 해제해서 비교 분석함.

**확인 과정**

- `Plg.bmp`, `Reset.bmp`, `Monster.bmp`의 실제 픽셀 데이터를 직접 읽어 확인 — 셋 다 정확한 순백색(255,255,255), 24비트 무압축 BMP로 파일 자체는 문제없음을 확인.
- `BmpMgr.cpp`, `Stage.cpp`, `Bitmap.cpp`를 모두 원본과 대조 — 전부 정상.
- `Monster.cpp`의 `Render()` 함수를 원본과 직접 비교한 결과, **원본에는 없는 `Rectangle(_DC, m_tRect.left, m_tRect.top, m_tRect.right, m_tRect.bottom);` 한 줄이 추가**되어 있던 것을 발견.

**원인**: `Rectangle` 함수는 기본 브러시가 흰색(WHITE_BRUSH)이라, 화면(`_DC`)에 흰 사각형을 먼저 그려놓음. 이후 `GdiTransparentBlt`가 흰색을 투명 처리해서 합성해도, 그 "투명 처리되어 그대로 남은 영역"이 바로 `Rectangle`이 미리 그려둔 흰 사각형이라 결과적으로 흰 테두리가 계속 보이는 것. Plg/Reset 비트맵이나 PlgBlt/GdiTransparentBlt/BitBlt 로직 자체는 모두 정상이었음.

**해결책**: `Rectangle` 줄을 삭제, 또는 디버그용으로 충돌 영역을 보고 싶다면 `NULL_BRUSH`로 바꿔서 채우기 없이 테두리만 그리는 방법.
