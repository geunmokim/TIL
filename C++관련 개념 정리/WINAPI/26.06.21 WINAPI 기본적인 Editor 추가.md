# WINAPI 기본적인 Editor 추가

## 1. Editor 프로젝트 추가 방법

- 학습 자료(오른쪽 압축파일)에 `Editor`라는 **새 프로젝트**가 같은 솔루션에 추가되어 있던 것을 확인
- 처음엔 "빈 프로젝트"로 추측했으나, 직접 압축을 풀어 다음을 확인한 결과 → **"Windows 데스크톱 응용 프로그램" 마법사**로 생성된 것이 맞음
  - `ReadMe.txt`에 "응용 프로그램 마법사로 만들었다"는 문구 존재
  - `stdafx.h/cpp`, `Resource.h`, `.rc`(메뉴 + About 다이얼로그 + 액셀러레이터) 등 마법사 표준 보일러플레이트가 전부 존재

### 새 프로젝트 추가 절차
1. 솔루션 우클릭 → **추가(Add) → 새 프로젝트(New Project)**
2. **"Windows 데스크톱 응용 프로그램"** 템플릿 선택 (C++ 카테고리)
3. 프로젝트 이름 입력 (예: `Editor`), 위치는 솔루션 폴더와 동일하게
4. 생성하면 `.cpp/.h`, `.rc`, `Resource.h`, 아이콘, `ReadMe.txt`까지 자동 생성됨

## 2. 마법사 옵션 — 빈 프로젝트 / 미리 컴파일된 헤더(PCH)

| 옵션 | 체크 여부 | 이유 |
|---|---|---|
| 빈 프로젝트 | **체크 해제** | 보일러플레이트(WinMain, WndProc, 리소스 등)가 이미 다 생성되어 있으므로 |
| 미리 컴파일된 헤더(PCH) | **체크함** | `stdafx.h` + `stdafx.cpp` 존재, `.vcxproj`에 `PrecompiledHeader=Create/Use` 설정 확인됨 |

### 비교 — 기존에 따라치던 프로젝트와의 차이
- 기존 프로젝트는 `framework.h`만 있고 PCH 관련 설정이 `.vcxproj`에 전혀 없음 → **PCH 미사용**으로 생성된 프로젝트
- `framework.h`라는 이름 자체가 "PCH 체크 해제하고 만들었을 때" 생성되는 결과물

### 핵심 결론
- **PCH는 빌드 속도 최적화 옵션일 뿐**, 프로그램의 동작이나 결과물(.exe)에는 전혀 영향 없음
- 빈 프로젝트로 직접 헤더를 만들어 빌드해도, 학습용 코드와 동일하게 동작하면 **완전히 정상**
- 오히려 마법사가 끼워주는 불필요한 코드(메뉴/About창 등) 없이 직접 다 타이핑하며 이해할 수 있어 학습 목적으로는 더 나은 방식일 수 있음

## 3. 공용 헤더 폴더 구조

### 구조
```
(솔루션 루트)
 ├─ Headers/              ← 공용 헤더, 어느 프로젝트 안에도 속하지 않음
 │   ├─ Include.h
 │   ├─ Define.h           (SAFE_DELETE 매크로 등)
 │   ├─ Enum.h
 │   ├─ Extern.h
 │   ├─ Function.h
 │   └─ Struct.h           (LINEPOS, LINEINFO 등)
 ├─ API_Framework/         ← 게임 프로젝트
 └─ Editor/                ← 에디터 프로젝트
```

### 연결 방법
- 각 프로젝트의 `.vcxproj`에 `<IncludePath>../Headers;$(IncludePath)</IncludePath>` 설정
- 프로젝트 폴더 기준 상대경로(`../Headers`)로 같은 폴더를 가리키게 하여 공유

### 실무와의 비교
- **데이터 포맷(구조체) 공유 자체**는 실무에서도 필수적이고 매우 흔함 (에디터가 만든 데이터를 게임이 읽어야 하므로)
- 하지만 **"폴더를 디렉터리 경로로 그냥 연결"하는 이 구체적인 방식**은 학습용/소규모 프로젝트에 흔한 약식 구현
- 실무 대형 프로젝트에서는 보통 공유 코드를 **별도의 정적 라이브러리(static lib) 프로젝트**로 분리하거나, 공유 모듈 시스템을 사용하는 경우가 더 일반적
- 핵심 원칙: **각자 다른 부분은 각자 구현, 공통으로 써야 하는 부분은 한 곳에만 두고 나머지가 설정(IncludePath 등)으로 참조**

### 이번 학습에서의 결정
- 이번엔 `Editor` 프로젝트 폴더 안에 필요한 헤더(`Define.h`, `Struct.h` 등)를 직접 만들어 넣고, "공용 헤더라는 방식도 있구나" 정도로 개념만 가져가기로 함
- 다음 학습 자료에서 Editor가 언로드되는 것으로 보여, 공용 폴더 구조를 유지할 실익이 적다고 판단

### 공용 헤더 폴더 전체 내용 (참고용)

**Define.h**
```cpp
#pragma once
#define WINCX 800
#define WINCY 600
#define SAFE_DELETE(p) if (p) { delete p; p = nullptr; }
#define OBJ_NOEVENT 0
#define OBJ_DEAD 1
#define PI 3.141592f
```

**Struct.h**
```cpp
typedef struct tagLinePos
{
	tagLinePos() { ZeroMemory(this, sizeof(tagLinePos)); }
	tagLinePos(float _x, float _y) : fX(_x), fY(_y) {}
	float	fX;
	float	fY;
}LINEPOS;

typedef struct tagLineInfo
{
	tagLineInfo() { ZeroMemory(this, sizeof(tagLineInfo)); }
	tagLineInfo(LINEPOS& _tLeftPos, LINEPOS& _tRightPos)
		: tLeftPos(_tLeftPos), tRightPos(_tRightPos) {}
	LINEPOS		tLeftPos;
	LINEPOS		tRightPos;
}LINEINFO;
```

**Function.h**
```cpp
template <typename T>
void Safe_Delete(T& _Dst)
{
	if (_Dst) { delete _Dst; _Dst = nullptr; }
}
```

## 4. Editor 코드 분석

### 충돌 판정 (`Collision_Line`)
- `LineMgr`에 구현은 있지만, `MainEditor`/`Editor.cpp` 어디에서도 호출되지 않음
- 게임 프로젝트의 `LineMgr` 코드를 복사해오면서 같이 따라온 **죽은 코드(dead code)**
- 에디터는 라인 생성/저장/로드만 담당, 충돌 판정은 게임 쪽 전용 기능

### `GetCursorPos` + `ScreenToClient`
- 항상 세트로 사용되는 패턴
- `GetCursorPos`: 마우스의 **모니터 전체(화면) 기준** 좌표
- `ScreenToClient`: 화면 기준 좌표 → **특정 창 내부 기준** 좌표로 변환
- 반대 방향 변환은 `ClientToScreen`

### 스크롤 보정 — `pt.x -= ScrollX`
- 화면(창)에 보이는 영역은 전체 맵의 일부일 뿐
- 창 내부 좌표를 맵 전체(월드) 기준 좌표로 다시 변환해야 정확한 위치에 라인이 저장됨
- `Set_ScrollX`(방향키 입력 시 호출)와 `Get_ScrollX`(마우스 클릭 시 호출)는 서로 다른 시점에 독립적으로 호출됨

### Get / Set 패턴
- **캡슐화(은닉화)** 목적: private 멤버 변수를 외부에서 직접 건드리지 못하게 막고, 정해진 함수를 통해서만 안전하게 접근하게 함
- `Get_` = 읽기 전용, `Set_` = 쓰기 전용 (이 코드만의 규칙이 아니라 객체지향 전반의 보편적 명명 관례)

### `ZeroMemory`로 bool 배열 초기화
- 메모리를 전부 0으로 채우면 `bool`은 확정적으로 `false`가 됨 (0 = false, 0이 아닌 값 = true)
- `bool` 배열을 `ZeroMemory`/`memset(..., 0, ...)`으로 초기화하는 것은 안전하고 흔한 패턴

### `Key_Down` / `Key_Up`
- 같은 키라도 **감지하려는 이벤트 타이밍**이 다름
  - `Key_Down`: false→true 전환(누르는 순간)을 감지, 그 순간 한 번만 true 반환
  - `Key_Up`: true→false 전환(떼는 순간)을 감지, 그 순간 한 번만 true 반환
- 코드 구조(두 개의 if)는 동일하고, 조건 순서와 `return true` 위치만 바뀐 것
- 키 종류와 함수 선택은 무관 — 같은 키에도 상황에 따라 `Key_Down`/`Key_Up`/`Key_Pressing` 중 적절한 것을 선택해서 사용

### `return`의 동작
- `return`을 만나는 즉시 함수 실행이 종료되고 호출한 곳으로 복귀
- 그 이하에 있는 코드는 조건과 무관하게 전부 무시됨 (if 블록 안에 있으면 그 조건이 참일 때만 조건적으로 종료)

### `Initialize()`를 별도로 두는 이유
- 생성자(메모리 할당)와 초기화(준비 작업)를 분리하는 설계 관례
- 지금은 한 줄(`m_bLineCheck = true`)이지만, 나중에 코드가 추가되고 복잡해질 자리를 미리 마련해두는 패턴
- 생성자는 객체 생성 시 단 한 번만 호출되지만, `Initialize()`는 필요시 다시 호출하여 리셋하는 용도로도 쓰일 수 있음

### 멤버 변수의 기본값
- `bool`, `int`, `float` 등 **기본 타입**은 멤버 변수로 선언만 하면 값이 자동으로 0/false가 되지 않음 (쓰레기값일 수 있음)
- class의 기본 접근 지정자(`private`)처럼 컴파일러가 보장하는 문법 규칙과는 다른 개념
- 따라서 생성자에서 반드시 명시적으로 초기화해야 함 (예: `m_bLineCheck(false)`)

### `typedef struct tagXXX {...} XXX;` 패턴
- C 언어 시절 관례의 잔재 (C에서는 구조체 사용 시 항상 `struct` 키워드를 붙여야 했기 때문에 typedef로 별명을 만들어 생략 가능하게 함)
- `tag` 접두사는 강제 문법이 아니라 "이건 typedef로 가려질 원본 구조체 이름"이라는 네이밍 관례
- 현대 C++에서는 `struct XXX {...};` 만으로 충분 (구조체 이름이 곧바로 타입명으로 쓰임)
- 게임업계는 레거시 코드/교재 스타일이 많이 남아있어 옛날 관례가 흔히 쓰임 — 정오답의 문제가 아니라 팀/프로젝트 컨벤션을 따르는 것이 중요

## 5. 실제 빌드/실행 트러블슈팅

### 시작 프로젝트 전환
- 솔루션 탐색기에서 원하는 프로젝트 우클릭 → **"시작 프로젝트로 설정"**
- 전체 빌드(`Ctrl+Shift+B`)는 솔루션 내 모든 프로젝트를 빌드하지만, 실행(F5)은 시작 프로젝트로 지정된 것 하나만 실행됨

### `g_hWnd` 링커 에러
- 오류: `확인할 수 없는 외부 기호 "struct HWND__ * g_hWnd"`
- 원인: `extern HWND g_hWnd;`는 **선언**일 뿐, 실제 **정의**(`HWND g_hWnd;`)가 메인 cpp에 빠져 있었음
- 해결: `Editor.cpp`(또는 해당 프로젝트의 메인 cpp)에 `HWND g_hWnd;` 정의 추가

### 창이 계속 깜빡이는 문제
- 원인: `CreateWindowW`의 너비/높이 인자에 `CW_USEDEFAULT, 0`을 잘못 입력하여 **높이 0짜리 창**이 생성됨
- 해결:
  ```cpp
  RECT rc = { 0, 0, WINCX, WINCY };
  AdjustWindowRect(&rc, WS_OVERLAPPEDWINDOW, FALSE);

  HWND hWnd = CreateWindowW(szWindowClass, szTitle, WS_OVERLAPPEDWINDOW,
     CW_USEDEFAULT, 0, rc.right - rc.left, rc.bottom - rc.top, nullptr, nullptr, hInstance, nullptr);

  if (!hWnd) return FALSE;

  g_hWnd = hWnd;   // 누락되어 있던 대입문도 추가
  ```

### 저장 실패 (`Line.dat`)
- 원인: `../Data/Line.dat`는 **.exe 실행 위치 기준 상대 경로**인데, 실제 `.exe`가 `Editor/x64/Debug/`에 생성되어 `../Data`가 `Editor/x64/Data/`를 가리키게 됨. 해당 폴더 자체가 존재하지 않아 실패
- 해결 방법 두 가지:
  1. **간단한 방법**: `Editor/x64/` 안에 `Data` 폴더를 직접 생성
  2. **정석적인 방법**: 프로젝트 속성 → 출력 디렉터리를 `$(SolutionDir)$(Platform)\$(Configuration)\`으로 통일하여, 솔루션 루트의 `Data` 폴더를 게임/에디터 프로젝트가 함께 공유하도록 설정

### API 학습용 실행 시 저장한 라인이 안 보이는 문제
- 원인: `API 학습용`의 `LineMgr::Initialize()`는 좌표 4개를 코드에 **하드코딩**해두고 있으며, `Line.dat`을 읽어오는 `ReadFile` 로직이 아직 구현되어 있지 않음
- Editor 쪽 문제가 아니라, 게임 프로젝트 쪽에 **아직 진행되지 않은 학습 단계**로 추정
- 직접 구현 시 필요한 형태 (참고):
  ```cpp
  void CLineMgr::Initialize()
  {
      HANDLE hFile = CreateFile(L"../Data/Line.dat", GENERIC_READ
          , NULL, NULL, OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, NULL);

      if (INVALID_HANDLE_VALUE == hFile)
          return;

      DWORD       dwByte = 0;
      LINEINFO    tTemp;
      while (true)
      {
          ReadFile(hFile, &tTemp, sizeof(LINEINFO), &dwByte, NULL);
          if (0 == dwByte)
              break;
          m_listLine.emplace_back(new CLine(tTemp.tLeftPos, tTemp.tRightPos));
      }
      CloseHandle(hFile);
  }
  ```
