# WINAPI 애니메이션 재생, 타일, 타일피킹

## 1. ScrollX 좌표 변환 공식의 출처

- `화면좌표 = 월드좌표 + ScrollX`(그리기), `월드좌표 = 화면좌표 - ScrollX`(마우스 클릭 좌표 계산)라는 공식이 "정해진 수학 법칙"인지, 아니면 "설계자가 정한 것"인지에 대한 질문
- **결론**: 절대적인 공식이 아니라, **"ScrollX를 어떤 방향(+/-) 의미로 쓰기로 정했는가"라는 설계 선택**에 따라 결정된 것
  - `Player::Key_Check()`에서 왼쪽 이동 시 `Set_ScrollX(+m_fSpeed)`, 오른쪽 이동 시 `Set_ScrollX(-m_fSpeed)`로 정한 것이 출발점
  - 단, 한쪽 방향(그리기)을 어떻게 정하든, **그 역방향(마우스 좌표 환산)은 수학적으로 자동으로 반대 부호가 되어야 함** — 이 "일관성"만큼은 임의로 정하는 게 아니라 자동으로 따라오는 결과
  - 부호 체계를 처음부터 반대로 설계해도 동일하게 동작하는 게임을 만들 수 있음 — 핵심은 "어느 쪽으로 정하든 일관성을 유지하는 것"

## 2. 타일맵(`CTileMgr`) 코드 분석

### for문 순서 — `TILEY`가 바깥쪽인 이유
- `for (i < TILEY) { for (j < TILEX) }` 구조에서 왜 `TILEY`가 바깥쪽인지에 대한 질문
- **결론**: 절대적 법칙이 아니라 **"한 줄(가로, TILEX개)을 다 채우고 다음 줄로 내려가는" 일반적인 2차원 격자 순회 관례** — 반대로 짜도 결과(전체 타일 집합)는 동일, 순서만 달라짐

### 타일 중심 좌표 계산식
- `(TILECX >> 1) + (j * TILECX)`가 무슨 식인지에 대한 질문
- **결론**: `j * TILECX`(j번째 칸이 시작하는 위치) + `TILECX >> 1`(타일 크기의 절반, 중앙으로 보정) = **j번째 칸의 정중앙 x좌표**를 구하는 식
- 이는 `CObj`가 항상 "중심점(fX, fY)" 기준으로 위치를 다루는 설계와 일치하는 패턴 (`Update_Rect()`에서 본 것과 동일한 원리)

### `CTileMgr` 생성자의 주석 해석
- `m_iSize(TILEX * TILEY)`, 주석으로 적힌 "+2", "+1은 부자연스러움", "Y값에 상관없는 것들 — UI, 타일", "그리는 순서는 배경,오브젝트,이펙트,UI순" 등에 대한 해석 요청
- 이 주석들은 **현재 코드에는 구현되지 않은, 향후 추가될 기능에 대한 메모(TODO)**로 확인됨:
  - "+2": 화면에 보이는 부분만 타일을 그리는 최적화를 할 때, 정수 나눗셈으로 인한 소수점 오차 때문에 화면 가장자리가 잘려 보이는 것을 방지하기 위해 여유분 2칸을 더 그리는 것
  - "+1은 부자연스러움": 스크롤이 움직이는 도중에는 여유 1칸으로는 부족해서 +2가 필요하다는 설명
  - "Y값에 상관없는 것들 — UI, 타일": 타일(배경)과 UI는 항상 정해진 순서(맨 뒤/맨 앞)로 그려지면 되고, y좌표에 따라 그리는 순서를 바꿀 필요가 없다는 의미
  - "배경,오브젝트,이펙트,UI 순으로 그리고, 이중 오브젝트만 Y값을 비교": Y-정렬(Y-sorting) 기법에 대한 계획 — 오브젝트(플레이어, 몬스터 등)만 y좌표가 클수록 나중에 그려서, 아래에 있는 캐릭터가 위에 있는 캐릭터보다 앞에 보이는 입체감을 구현하려는 의도
- 결론: 지금 코드만 봐서는 이해가 안 되는 게 당연하며, 다음 학습 단계에서 실제 구현이 나올 것으로 추정

## 3. 애니메이션 프레임 시스템(`Player.cpp`, `Obj::Update_Frame`) 심층 분석

### `Update_Frame()`의 시간 체크 조건이 항상 거짓이 아닌가에 대한 의문
- `if (dwTime + dwDelayTime < GetTickCount())` 조건이, 갱신 직후엔 항상 거짓이라 "적용이 안 되는 것 아니냐"는 질문
- **결론**: 갱신된 그 순간만 보면 거짓이 맞지만, `GetTickCount()`는 호출할 때마다 현재 시각이 계속 증가하는 값이라 **시간이 흐르면 결국 조건이 참이 됨** — 알람을 맞춘 직후엔 안 울리지만 시간이 지나면 울리는 것과 같은 원리
- 매 프레임(예: 16ms)마다 반복 호출되다가, `dwDelayTime`만큼 시간이 누적되면 조건이 참이 되어 프레임 전환 + 시각 리셋이 일어남

### `m_tFrame.iStartX`의 정확한 의미
- "프레임의 시작점"이라는 표현이 정확한지에 대한 질문
- **정확한 표현으로 합의됨**: "지금 이 순간, 스프라이트 시트(여러 동작 컷이 가로로 나열된 한 장의 이미지)에서 몇 번째 칸을 잘라서 보여주고 있는지를 나타내는 인덱스(번호)"
- 비유: 만화책 한 페이지(전체 BMP, 안 바뀜)에서 "지금 손가락으로 짚고 있는 칸 번호"가 `iStartX`이며, `Update_Frame()`이 이 손가락을 옆으로 옮기는 역할

### `dwDelayTime`이 작을수록 애니메이션이 빨리 재생되는가
- "딜레이가 작은 모션일수록 같은 시간동안 더 많이 재생되는 것 아니냐"는 질문
- **정확함**: `dwDelayTime`이 작을수록 같은 1초 동안 프레임이 더 자주 전환되고, 결과적으로 전체 애니메이션 사이클(루프)도 더 많이 반복됨 — 영상 필름을 빨리 넘기는 것과 동일한 원리

### 스프라이트 시트 이미지(Player_DOWN.bmp, 1200x1000) 분석
- 업로드한 스프라이트 시트 이미지(5줄 구성: 1번째 4컷, 2번째 6컷, 3번째 6컷, 4번째 2컷, 5번째 4컷)를 보고 `WALK`(iEndX=6)가 2번째 또는 3번째 줄에 해당한다는 추론
- 실제 픽셀 계산(1200÷200=6칸 가로 확인, 1000÷5줄=한 줄당 200px)으로 검증한 결과:
  - `enum STATE{IDLE, WALK, ATTACK, HIT, DEAD, END}`에서 `WALK=1`
  - `iStateY * 400` 계산식 기준, `WALK`는 `1*400=400px` 위치, 즉 **3번째 줄(6컷)**을 정확히 가리킴
  - `IDLE`(0)은 0px(1~2번째 줄 영역), `ATTACK`(2)은 800px(5번째 줄 영역)에 대응
  - 이 "한 단위=400px(두 줄 분량)" 설계 때문에 한 단위가 정확히 200px(한 줄)씩 떨어지지 않는 위치(예: 2번째 줄, 200px)는 `iStateY`만 바꿔서는 표현 불가능하며, `Render()`의 곱하는 수(400)도 같이 200으로 바꿔야 함

### `iStartX = 0`과 `iEndX`의 의미 재확인
- `Frame_Check()`의 `iStartX = 0`(상태 전환 시 초기화)와 `Update_Frame()`의 `iStartX = 0`(애니메이션 끝났을 때 순환 리셋)이 서로 다른 목적의 코드라는 점 재확인 — 전자는 "새 애니메이션은 항상 처음부터", 후자는 "끝까지 돌았으니 다시 처음으로 순환"
- `iStartX=0`은 "애니메이션의 첫 번째 컷"이라는 의미상 자연스럽게 0이 되는 것이며, 의도적으로 다른 시작점(예: 3)을 줄 수도 있음
- `iEndX`는 **BMP 이미지에서 해당 동작이 실제로 몇 컷으로 나뉘어 그려졌는지를 그대로 코드에 반영한 값** — 코드가 임의로 정하는 게 아니라 리소스 구성과 정확히 일치해야 하는 값

## 4. `Scroll_Lock()` 코드 분석

```cpp
void CScrollMgr::Scroll_Lock()
{
	if (0 <= m_fScrollX)
		m_fScrollX = 0.f;
	if (WINCX - (TILEX * TILEY) >= m_fScrollX)
		m_fScrollX = WINCX - (TILEX * TILECX);
}
```

- 코드의 목적(카메라가 맵 바깥의 빈 공간을 보여주지 않도록 제한)에 대한 이해 확인
- 첫 번째 if문: ScrollX가 0 이상이 되면 0으로 고정 — 맵의 왼쪽 한계를 막는 것 (정확한 해석으로 확인됨)
- 두 번째 if문: 조건문에는 `TILEX * TILEY`, 대입문에는 `TILEX * TILECX`로 서로 다른 식이 쓰인 것은 **오타로 추정** — 올바르게는 둘 다 `TILEX * TILECX`(맵 전체의 가로 길이)여야 함
- 최종 정리된 이해: **맵은 크기에 한도가 있고, 플레이어가 계속 이동하면 스크롤도 따라가려 하므로 언젠가는 맵 데이터가 끝나는 지점을 넘어 빈 공간이 보일 수밖에 없는데, `Scroll_Lock()`이 이를 막는 안전장치** — 정확한 이해로 확인됨

### 왜 가로(X)만 제한하는지에 대한 질문
- 세로(Y) 방향으로도 같은 방식의 제한을 거는 게 가능한지에 대한 질문
- **결론**: `ScrollMgr`에 `m_fScrollY` 자체가 존재하지 않으며, 이 게임은 처음부터 **가로(사이드) 스크롤 전용 구조**로 설계되어 있어서 세로 제한이 의미가 없는 것
- "가로로 잡아야 한다"는 절대적 규칙이 아니라, **이 게임의 설계(가로 스크롤 전용 구조)에 맞춘 선택일 뿐**이며, 세로 스크롤이 있는 다른 게임(탑다운 등)이라면 동일한 방식으로 세로 버전도 구현 가능하다는 것으로 합의됨

## 5. 메뉴 화면에서 Start 버튼을 눌러도 버튼이 안 사라지는 문제

- Start 버튼을 눌러도 Start/Edit/Exit 버튼 이미지가 화면에 그대로 남아있는 문제
- **원인**: `SceneMgr::Scene_Chaange()`(또는 `Release()`)가 씬 객체(`CMyMenu` 인스턴스) 자체만 삭제하고, 그 씬이 `ObjMgr`에 등록해두었던 버튼 객체들(`OBJID::UI`)은 별도로 처리하지 않아 그대로 남아있었기 때문
- `ObjMgr`은 "버튼이든 플레이어든 다 동일하게 취급해서 매 프레임 그리는" 범용 매니저라서, 명시적으로 제거 요청을 하지 않으면 계속 그려짐
- **해결**: `CMyMenu::Release()`에 메뉴가 등록했던 UI 오브젝트들을 명시적으로 `ObjMgr`에서 제거하는 코드를 추가하여 해결됨 (실제로 적용하여 해결 확인됨)

### 에딧(Edit) 화면에서는 반대로 버튼이 안 보이는 현상에 대한 추가 질문
- 메뉴에서는 버튼이 남아 보였는데, 에딧 화면에서는 타일만 보이고 버튼이 안 보이는 것에 대해, "에딧이 타일로 화면을 도배해서 버튼 위에 덮어쓴 것이냐"는 추론
- 실제 `CMyEdit::Render()` 코드 확인 결과:
  ```cpp
  void CMyEdit::Render(HDC _DC)
  {
      CTileMgr::Get_Instance()->Render(_DC);
  }
  ```
- **정정된 원인**: "타일이 버튼을 덮어써서 안 보이는 것"이 아니라, **`CMyEdit::Render()`에 `CObjMgr::Get_Instance()->Render(_DC)` 호출 자체가 없어서, 버튼 객체가 `ObjMgr`에 남아있어도 화면에 그려질 기회 자체가 없었던 것**으로 확인됨
- 결과(버튼이 안 보임)는 추측과 같았지만, 원인(덮어쓰기 vs 그리기 호출 자체의 부재)은 다름을 구분함

## 6. `Player.cpp` 원본 vs 본인 코드 비교 디버깅 (3차례에 걸친 트러블슈팅)

### 1차 문제 — 플레이어가 너무 작고, 좌우 이동 시 이미지가 안 보임
- **원인 1**: `m_tInfo.iCX/iCY`를 100으로 설정 (원본은 200) → 절반 크기로 보임
- **원인 2**: `Insert_Bmp`로 등록한 키 이름(`"Player_Left"`, `"Player_Right"`, 첫 글자만 대문자)과 `Key_Check()`에서 찾는 키 이름(`"Player_LEFT"`, `"Player_RIGHT"`, 전체 대문자)의 **대소문자가 일치하지 않아** `lstrcmp` 비교가 실패, `Find_Image`가 `nullptr`을 반환하여 이미지가 안 보임 — 이전에 다뤘던 "Back/Bacak" 오타 문제와 동일한 종류의 버그
- **해결**: `iCX/iCY`를 200으로, `Key_Check()`의 키 이름 대소문자를 등록된 키와 일치시킴

### 2차 문제 — 방향을 바꿔도 여전히 Down 이미지로 보이고, 이동 시 제자리에서 걸리는 느낌
- **원인 1**: `Initialize()`에서 `m_eCurState = IDLE` 대신 `m_ePreState = IDLE`로 오타가 있어, `m_eCurState`가 초기화되지 않은 쓰레기 값을 가져 `Frame_Check()`의 상태 비교 로직이 예측 불가능하게 동작
- **원인 2**: `Late_Update()` 안에서 `Key_Check()`, `Jumping()`, `OffSet()`, 그리고 `Update()` 자체까지 중복 호출하고 있어, 한 프레임 안에서 이동/스크롤 보정 로직이 여러 번 실행되어 좌표 계산이 꼬임 (원본은 `Late_Update()`가 비어있음)
- **해결**: `m_ePreState` → `m_eCurState`로 수정, `Late_Update()`를 비워둠

### 3차 문제 — 오른쪽/왼쪽 이동이 전혀 안 되는 것처럼 보임
- 코드 확인 결과 `Render()`에서 `Find_Image(_T("Player_Down"))`으로 이미지 키가 강제로 고정되어 있었던 점(디버깅용으로 임시 변경된 상태로 추정)을 짚었으나, 이는 그림(이미지)에만 영향
- **진짜 원인**: `OffSet()`과 `Scroll_Lock()`의 상호작용 — 맵 중간 구간에서는 카메라(`OffSet`)가 플레이어를 따라가며 화면 중앙에 고정시키기 때문에, 실제로는 `m_tInfo.fX`(좌표)가 정상적으로 바뀌고 있어도 화면상으로는 캐릭터가 안 움직이는 것처럼 보임. 맵 가장자리에 도달해 `Scroll_Lock`이 카메라를 멈추면, 비로소 캐릭터의 실제 이동이 화면에 드러나 보임
- 사용자가 직접 "Scroll_Lock 때문에 그런 것이고, 맵 끝까지 이동시키니 이동이 보인다"는 것을 인지하며 정확히 원인을 파악함 — 버그가 아니라 스크롤 카메라 시스템의 정상 동작으로 확인됨
- 최종적으로 모든 문제가 해결됨 (대소문자 오타, PreState/CurState 혼동, Late_Update 중복 호출, Render의 임시 코드, Scroll_Lock으로 인한 시각적 착시까지 전부 해결)
