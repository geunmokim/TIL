# WINAPI Offset 추가와 스크롤 구현

## 1. ScrollX 값을 Player 안에 따로 두지 않고 ScrollMgr로 분리한 이유

- `Set_ScrollX`가 매개변수 값을 계속 누적하는 함수인데, 굳이 `Player::Key_Check()`에서 별도 변수로 안 하고 `ScrollMgr`를 통해서 처리하는 이유에 대한 질문
- **핵심 이유**: 단순히 "값을 누적해야 해서"가 아니라, **`Player` 말고도 `Line`, `Monster` 등 화면에 그려지는 다른 모든 객체들이 똑같은 스크롤 값을 읽어서 좌표 보정에 써야 하기 때문**
- 만약 `Player` 안에만 변수로 두면, 다른 클래스들이 그 값을 가져올 방법이 없고, 객체마다 따로 값을 들고 있으면 동기화 문제가 생김
- 결론: 여러 객체가 공통으로 참조해야 하는 값이라서, 한 곳(`ScrollMgr` 싱글톤)에 모아두고 모두가 그곳을 보게 만든 것 (공용 헤더와 같은 발상)

## 2. 싱글톤 패턴의 정의 — "공용으로 쓰니까"가 아니라 "코드로 강제해서 하나만 존재"

- "공용으로 사용해서 같은 변수를 안 만드니까 싱글톤이 된 것"인지, 아니면 "다른 곳에서 똑같은 걸 못 만들게 코드로 막아둔 것"인지 질문
- **정답은 후자**: 싱글톤 패턴의 본질은 **"이 클래스의 인스턴스는 프로그램 전체에서 딱 하나만 존재하도록 코드로 강제하는 것"**
- 구현 방법:
  - 생성자를 `private`으로 선언 → 외부에서 `new CScrollMgr` 직접 호출 자체를 컴파일 단계에서 차단
  - `Get_Instance()`라는 `public static` 함수를 유일한 통로로 제공 → 이미 만들어졌으면 기존 것 반환, 없으면 그 안에서 딱 한 번만 생성

## 3. 생성자를 private으로 만든 게 정확히 무엇을 막는 것인지

- "초기화 리스트에 접근 못 하게 막은 것"이라는 표현에 대한 정정
- 정확히는 **초기화 리스트만이 아니라 생성자 전체(객체를 new로 만드는 행위 자체)**가 막히는 것
- `private` 생성자는 외부에서 "이 함수가 안 보이는 것"처럼 작동해서, `new CScrollMgr`라는 행위 자체를 못 하게 차단함

## 4. class 전체와 그 안의 멤버 변수가 싱글톤에 의해 하나만 존재하게 되는지

- `class CScrollMgr` 전체가 싱글톤 패턴이 적용된 것이고, 그 안의 멤버 변수(`m_fScrollX`)도 전체에서 하나만 생성되는 것인지 질문
- **둘 다 맞음**: 객체 자체가 하나뿐이므로, 그 객체에 속한 멤버 변수도 자연스럽게 메모리상 하나만 존재하게 됨

## 5. 멤버 변수가 공유/누적되는 게 static 변수와 같은 원리인지

- `m_fScrollX`가 공유되고 누적되는 게 "static 변수처럼" 동작하는 것인지 질문
- **결과는 같지만 원리는 다름**:
  - `m_fScrollX`는 `static`이 안 붙은 **평범한 멤버 변수**임
  - 진짜 `static`이 붙은 건 `static CScrollMgr* m_pInstance;`(객체를 가리키는 포인터)
  - `m_fScrollX`가 하나뿐인 이유는 "static이라서"가 아니라, **그 변수가 속한 객체(CScrollMgr 인스턴스) 자체가 싱글톤이라 하나뿐이기 때문**
- 비유: static 변수 = "동네에 우체통이 원래 하나"; 싱글톤 = "집을 한 채만 짓게 강제해서, 그 집 안의 가구도 자연히 하나씩"

## 6. Get_Instance()와 static m_pInstance가 세트로 필요한 이유

- `Get_Instance()` 함수 혼자서 객체를 하나만 만드는 거라고 생각했는데, `static CScrollMgr* m_pInstance`까지 합쳐져야 그런 건지 질문
- **정답: 둘이 세트로 필요함**
  - `static m_pInstance` = 만들어진 객체의 주소를 기억하는, 클래스 전체에서 단 하나뿐인 "상자"
  - `Get_Instance()` = 그 상자를 확인하고 관리하는 로직
  - `m_pInstance`가 static이 아니라면 "객체별로 따로 갖는 변수"가 되어버려, static 함수인 `Get_Instance()` 안에서는 애초에 접근할 방법이 없어짐 (어느 객체의 변수인지 알 수 없으므로)

## 7. static의 "값 공유" 성질 덕분에 m_pInstance가 명확히 식별되는 것인지

- static 변수의 "값이 공유된다"는 성질 때문에 `m_pInstance`가 하나로 유지되고, 그래서 `Get_Instance()`가 "어떤 m_pInstance인지" 헷갈리지 않고 다룰 수 있는 것인지 질문
- **정확히 맞음**: `m_pInstance`가 static이라 "하나"라는 게 먼저 보장되어 있어야, `Get_Instance()`가 헷갈림 없이 그 하나를 안전하게 확인·생성·반환할 수 있음

## 8. Get_Instance()에 static을 붙인 이유 — private 생성자 때문인지 질문 (오개념 정정)

- "private로 생성자 접근을 막아버려서 Get_Instance()에 static을 써야 하는 것"이라는 추측
- **정정**: 이 인과관계는 틀림. 둘은 별개의 이유로 각각 필요함
  - `private` 생성자: 외부에서 마음대로 `new`해서 여러 개 만드는 것을 방지 (static과 무관)
  - `Get_Instance()`가 static인 이유: "객체를 얻기 위한 함수인데, 그 함수를 호출하려면 이미 객체가 있어야 한다"는 닭과 달걀 모순을 피하기 위해 — 객체 없이도(클래스 이름으로) 호출 가능해야 하므로 static이 필수

## 9. 최종 확인 — Get_Instance()가 static이 아니면 "객체.Get_Instance()"라는 이상한 형태가 되는지

- 최종적으로 "Get_Instance() 자체가 객체를 생성하기 위한 함수인데, static이 아니면 객체.Get_Instance()라는 모순된 형태가 된다"는 이해로 정리
- **정확히 확인됨**: 객체를 얻기 위해 부르는 함수인데, 그 함수를 부르기 위해 이미 객체가 필요하다는 모순(닭과 달걀 문제)이 발생하며, `static`을 붙여 클래스 이름으로 직접 호출(`CScrollMgr::Get_Instance()`) 가능하게 함으로써 이 모순을 해결함

## 10. LineMgr의 Load_Line() — 하드코딩에서 파일 입출력으로 전환

- `Load_Line()`만 보면 "선을 그리는 코드가 없는 것 아니냐"는 질문에서 시작
- 정리된 내용:
  - `Load_Line()`은 파일(`Line.dat`)에서 좌표를 읽어와(`ReadFile`) `CLine` 객체를 생성하며 **`m_tInfo`에 좌표를 저장**하는 역할만 함 (데이터 준비 단계)
  - 화면에 실제로 그리는 건 `CLineMgr::Render()` → 각 `CLine::Render()`가 맡음 (이미 저장된 `m_tInfo`를 그대로 가져다 `MoveToEx`/`LineTo`로 그림)
  - 즉 "좌표를 받아오는 과정"은 `Render()`가 아니라 `Load_Line()`(과 `CLine` 생성자) 시점에 이미 끝나 있는 것
- 예전 방식(좌표 하드코딩, 코드에 직접 `LINEPOS tPos[4] = {...}`)과 비교:
  - 예전: 좌표가 소스 코드에 박혀있어 수정하려면 재컴파일 필요
  - 지금: 좌표가 `Line.dat` 파일에 있어, Editor에서 다시 그려 저장만 하면 게임이 새 맵을 그대로 반영 — Editor와 게임이 데이터를 공유하는 흐름이 실제로 완성된 형태

## 11. Offset 이미지출력작업중 단계 — 빌드/실행 트러블슈팅

### 사각형 테두리가 보이는 문제
- `Rectangle(m_hDC, 0, 0, WINCX, WINCY)`로 매 프레임 배경을 덧칠하는 기법 자체는 원본과 동일하게 사용 중이었으나, 본인 코드에서만 테두리가 보임
- **원인**: `CreateWindowW` 호출 시 너비/높이 인자에 `CW_USEDEFAULT, 0`을 그대로 사용 — 클라이언트 영역이 `WINCX`/`WINCY`(800x600)와 정확히 일치하지 않게 생성됨
- **해결**: `RECT rc = {0,0,WINCX,WINCY}; AdjustWindowRect(&rc, WS_OVERLAPPEDWINDOW, FALSE);`로 클라이언트 영역을 정확히 800x600에 맞춘 뒤, 그 결과(`rc.right-rc.left`, `rc.bottom-rc.top`)를 `CreateWindowW`에 전달 → `Rectangle`의 경계가 창 가장자리와 겹쳐서 테두리가 안 보이게 됨

### 카메라가 플레이어를 따라오지 않는 문제 → Player::Render에 스크롤 미적용
- `OffSet()` 로직 자체(`ScrollMgr`의 `Set_ScrollX` 호출)는 정상 작동 중이었음
- **원인**: `CPlayer::Render()`에서 `iScrollX` 값을 가져오긴 했지만, 실제 `Rectangle()` 호출 시 그 값을 더하지 않고 있었음
  ```cpp
  // 문제였던 코드
  Rectangle(_DC, m_tRect.left, m_tRect.top, m_tRect.right, m_tRect.bottom);
  ```
- **해결**: `left`와 `right`에 `+ iScrollX`를 추가
  ```cpp
  Rectangle(_DC, m_tRect.left + iScrollX, m_tRect.top, m_tRect.right + iScrollX, m_tRect.bottom);
  ```

### 플레이어가 오른쪽 키를 누르면 위로 솟구치는 문제 → Line(라인)에 스크롤 미적용
- Player에 스크롤 보정을 추가한 후 새로 나타난 증상: 오른쪽 키를 누르면 캐릭터가 Y축 위로 솟구치고, 경사가 끝나는 지점에서 멈춤
- 처음에는 "경사면을 타고 올라가는 정상 동작 아니냐"고 의심했으나, 스크린샷 비교로 실제 원인이 다름을 확인
- **진짜 원인**: `Line::Render()`에 스크롤 보정(`+ iScrollX`)이 전혀 적용되어 있지 않았음 (`#include "ScrollMgr.h"` 자체가 없었음)
  ```cpp
  // 문제였던 코드 (본인 Line.cpp)
  MoveToEx(_DC, (int)m_tInfo.tLeftPos.fX, (int)m_tInfo.tLeftPos.fY, nullptr);
  LineTo(_DC, (int)m_tInfo.tRightPos.fX, (int)m_tInfo.tRightPos.fY);
  ```
- **결과적으로 벌어진 문제**: 플레이어는 스크롤 보정이 적용되어 화면에서 카메라를 따라 이동했지만, 라인(지형)은 스크롤 보정 없이 항상 절대좌표(고정된 위치)로 그려짐 → 두 객체가 서로 다른 좌표계로 그려지면서 시각적으로 어긋나 보임 (실제 논리적 위치/충돌 계산은 둘 다 정상이었음)
- **해결**: `Line.cpp`에도 `ScrollMgr`를 include하고, `Render()`에서 동일하게 `+ iScrollX`를 적용
  ```cpp
  #include "ScrollMgr.h"   // 추가

  void CLine::Render(HDC _DC)
  {
      int iScrollX = (int)CScrollMgr::Get_Instance()->Get_ScrollX();   // 추가
      MoveToEx(_DC, (int)m_tInfo.tLeftPos.fX + iScrollX, (int)m_tInfo.tLeftPos.fY, nullptr);
      LineTo(_DC, (int)m_tInfo.tRightPos.fX + iScrollX, (int)m_tInfo.tRightPos.fY);
  }
  ```
- **핵심 결론**: 스크롤(카메라 추적) 기능을 구현할 때는, 화면에 그려지는 **모든 객체(Player뿐 아니라 Line 등 전부)**가 동일하게 `+ iScrollX` 보정을 받아야 한다. 하나라도 빠지면 그 객체만 카메라를 따라가지 않고 고정된 것처럼 보여, 서로 좌표계가 어긋나는 시각적 버그가 발생함
