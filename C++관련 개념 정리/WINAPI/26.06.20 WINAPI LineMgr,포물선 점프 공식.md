1. struct vs class

Set_Pos(float, float) 같은 **일반 멤버 함수(setter)**는 객체 생성 후 값을 바꾸는 용도, tagLinePos(float, float) : fX(_x), fY(_y) {} 같은 건 생성자이며 : fX(_x), fY(_y)는 멤버 초기화 리스트(객체 생성 시점에 바로 초기화)
C++의 struct는 class와 기능적으로 동일함 (생성자, 멤버함수, private, 상속, virtual까지 다 가능). 유일한 차이는 기본 접근 지정자(struct=public, class=private)
그럼에도 관습적으로 구분해서 씀:

순수 데이터 묶음(POD) → struct (생성자/소멸자 정도까지는 자연스러움)
캡슐화 + 행동(virtual, 다형성, private+getter/setter) → class
LINEPOS, LINEINFO처럼 "값을 묶어서 초기화 편하게 해주는" 생성자는 여전히 struct스러운 범위


이는 컴파일러 입장에서는 의미 없는 구분이지만, 협업 시 코드의 의도를 빠르게 전달하기 위한 암묵적 관습이라는 결론

2. CLine / CLineMgr 구조 분석

의존 관계 방향: LineMgr.h가 Line.h를 포함(매니저가 개별 객체를 알아야 함), 반대로 Line.h는 LineMgr.h를 포함하지 않음(개별 객체는 매니저를 몰라도 됨) → 단방향 의존성이 좋은 설계 패턴
역할 분리: CLine은 "두 점을 받아 선 하나를 그리는" 단일 기능만 담당, CLineMgr은 CLine을 몇 개 만들지·언제 그릴지를 결정하는 관리자(지휘) 역할
CLine 생성 시 new CLine(tPos[0], tPos[1]) → 생성자 인자(_tLeftPos, _tRightPos, 참조 전달) → 멤버 초기화 리스트(m_tInfo(_tLeftPos, _tRightPos)) → 내부적으로 LINEINFO 생성자 호출 → tLeftPos, tRightPos 채워짐, 이렇게 생성자 호출이 체인처럼 연쇄되며 한 줄로 압축됨
m_listLine(list<CLine*>)에 포인터들을 저장 → CLineMgr::Render()에서 for (auto& pLine : m_listLine) pLine->Render(_DC);로 순회하며 각 객체의 Render()를 호출 → 선 3개가 화면에 그려지는 흐름
이 패턴이 CMainGame → CLineMgr → CLine처럼 상위 계층이 하위 계층에게 책임을 위임하는 구조로 한 단계 위에서도 반복됨을 확인 (CMainGame은 CLineMgr에게 "그려"라고만 지시하고 세부사항은 모름)

3. 빌드/컴파일 순서

빌드는 컴파일(.cpp→.obj) → 링크(.obj들→.exe) 두 단계
.cpp 파일들 사이의 컴파일 순서, .obj 파일들의 링크 순서는 결과에 거의 영향 없음 (각 파일이 자기완결적이고, 링커는 누락 여부만 검사)
실질적으로 중요한 건 한 .cpp 파일 안에서 #include/코드가 위에서 아래로 읽히는 순서 — #include는 그 자리에 헤더 내용을 그대로 복사+붙여넣기하는 것과 같아서, 어떤 타입/함수를 쓰기 전에 그게 선언된 헤더가 먼저 포함되어 있어야 함
컴파일러는 코드를 읽다가 모르는 타입/함수를 만나면, 그 시점까지 포함된 헤더들 안에서 선언을 찾는 방식으로 동작 — 이는 사람이 코드를 읽다가 모르는 게 나오면 해당 헤더를 열어 확인하는 것과 동일한 방식

4. CLineMgr::Collision_Line 함수 분석

_y가 float*(포인터)인 이유: 함수의 반환값(bool)은 이미 "충돌 여부"를 알려주는 데 쓰이고 있어서, 추가 정보(충돌 지점의 y좌표)는 포인터를 통해 호출자가 마련해둔 메모리 공간에 직접 써서 전달하는 구조 (출력 매개변수 패턴)
함수의 설계 의도: 단순히 "충돌했는지"만이 아니라, 기울어진 라인에서도 **그 x좌표에 정확히 대응하는 y좌표(높이)**까지 알아내기 위함 — 학습용 코드에서 실제로 사각형이 라인을 타고 움직이는 동작으로 확인됨
x좌표는 캐릭터가 스스로 결정하는 값(읽기만 함)이라 값으로 받고, y좌표는 함수가 계산해서 돌려줘야 하는 값이라 포인터로 받는 것 — 만약 _x도 포인터로 바꾸면 문법적으로는 가능하지만, 함수 안에서 _x를 바꾸는 로직이 없다면 불필요하고 가독성도 떨어짐

5. C++의 "값 하나만 반환" 규칙과 포인터의 관계

return은 C++ 언어가 보장하는 정식 반환 통로로, 정확히 하나의 값만 가능
포인터를 통해 *_y = 값;처럼 쓰는 것은 반환이 아니라 단순 대입(assignment) — 호출자가 미리 마련해둔 메모리 공간을 함수가 대신 수정하는 행위이지, 언어가 말하는 "반환"의 범주에 속하지 않음
그래서 "함수는 값 하나만 반환한다"는 규칙은 깨지지 않음 — 포인터 전달은 그 규칙이 다루는 대상 자체가 아님
다른 언어(Python, Go)는 여러 값을 동시에 반환하는 문법을 언어 차원에서 지원하지만, 내부적으로는 결국 튜플 등으로 묶이는 경우가 많음. C++도 std::tuple/struct로 묶으면 유사한 효과를 낼 수 있음

6. 출력 매개변수(포인터) vs tuple/struct — 업계 관행

검색으로 확인한 결과: Google C++ Style Guide는 "선택적 출력"(조건부로만 유효한 값, 지금의 _y처럼)에는 포인터 사용을 정당한 패턴으로 인정
C++ Core Guidelines는 출력값이 여러 개일 경우 포인터를 여러 개 늘어놓기보다 tuple/struct로 묶어 반환하는 걸 권장
게임 클라이언트 코드(Win32 API, DirectX 등)에서는 포인터 출력 매개변수 패턴이 매 프레임 호출 비용을 줄이는 이유 등으로 여전히 흔하게 쓰임
결론: 지금 학습 중인 코드 스타일(포인터 방식) 그대로 모작에 써도 무방하며, 게임 클라이언트 분야 특성상 자연스러운 선택. 다만 출력값이 2개 이상으로 늘어나면 tuple/struct를 고려할 수 있음

7. Collision_Line의 경계값 비교 (<= vs >)

tLeftPos.fX <= _x && tRightPos.fX > _x — 왼쪽 경계는 포함, 오른쪽 경계는 미포함하는 반열린 구간(half-open interval) 설계
이렇게 하는 이유: 라인이 서로 끝점을 공유하며 이어질 때, 양쪽 다 포함(<=/>=)으로 하면 경계점에서 두 구간이 동시에 조건을 만족해 리스트 순서에 따라 결과가 달라지는 모호함이 생김 (특히 기울기가 다른 구간이 만나는 지점에서 위치가 끊기거나 떨리는 버그로 이어질 수 있음)
이건 WinAPI의 고유 규칙이 아니라, 구간을 겹치지 않게 나누는 일반적인 프로그래밍 기법 (C++ STL의 begin()/end() 반복자 관례도 동일한 원리)
다만 이 설계의 한계: 라인이 이어지지 않는 맨 마지막 구간의 오른쪽 끝점에서는 받아줄 다음 구간이 없어, 그 정확한 지점에서 충돌 판정이 누락될 수 있는 미세한 여지가 있음

8. CPlayer의 점프 코드 분석 (Jumping(), Key_Check())

포물선 점프 공식: m_tInfo.fY = m_fJumpY - (m_fJumpPower * m_fJumpTime - 0.5f * 9.8f * m_fJumpTime * m_fJumpTime)

9.8은 실제 중력가속도, 0.5는 등가속도 운동 공식의 계수 — 둘 다 물리학적으로 고정된 값
m_fJumpPower(점프 세기)는 개발자가 게임 감각에 맞춰 임의로 정한 값
이 공식의 정식 명칭은 "등가속도 운동 공식" 또는 "포물선 운동 공식"


m_fJumpY를 따로 두는 이유: m_tInfo.fY는 이 식의 결과로 매 프레임 덮어써지는 값이라 기준점으로 쓰면 안 되고, m_fJumpY는 점프 시작 시점에 고정된 기준점(앵커) 역할을 해야 공식이 매번 올바르게 계산됨
오타 발견: (m_fJumpPower * m_fJumpTime = 0.5f * ...)에서 =가 들어간 건 -의 오타. 곱셈 결과(임시값, non-lvalue)에 =로 대입하려 해서 컴파일 에러(l-value 에러) 발생 원인이었음
Key_Check()의 if (m_bJump) m_fJumpY = m_tInfo.fY; 부분도 !m_bJump여야 했던 오타였음 — 점프 중이 아닐 때만 현재 위치를 기준점으로 저장해야 의미가 맞음

9. 게임에서 실제 물리/수학 공식을 가져다 쓰는 다른 사례들
