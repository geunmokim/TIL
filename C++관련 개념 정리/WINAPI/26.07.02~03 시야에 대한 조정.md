# 스톤샤드 턴제 RPG 모작 — 기술 구현 노트

## 1. 화살 방향 시스템 (Arrow Direction)

히트스캔 방식으로 구현된 화살이 타겟 칸 위치에만 배치되다 보니, 몬스터가 어느 방향에서 쐈든 상관없이 항상 칸의 왼쪽 모서리에 고정되어 보이는 문제가 있었다. 타일 크기(64px)와 스프라이트 크기(21px) 차이로 인해 `m_tRect.left`를 그대로 쓰면 칸 좌상단 기준으로 그려지기 때문이었다. 또한 화살촉 방향도 원본 스프라이트(오른쪽 향함) 한 장뿐이라 몬스터 위치와 무관하게 항상 같은 방향을 향하고 있었다.

`Arrow` 클래스에 `Set_Direction(srcPos, targetPos)`을 추가해 몬스터(발사 위치)와 플레이어(타겟) 간의 col/row 차이로 방향을 계산하고, 칸 내 오프셋과 반전 여부를 결정하도록 수정했다.

```cpp
void Set_Direction(const GRIDPOS& _startPos, const GRIDPOS& _targetPos)
{
    int iColDiff = _startPos.col - _targetPos.col;
    int iRowDiff = _startPos.row - _targetPos.row;
    int iMaxOffsetX = (TILECX - 21) / 2;
    int iMaxOffsetY = (TILECY - 21) / 2;

    m_iOffsetX = (iColDiff > 0) ? iMaxOffsetX : (iColDiff < 0) ? -iMaxOffsetX : 0;
    m_iOffsetY = (iRowDiff > 0) ? iMaxOffsetY : (iRowDiff < 0) ? -iMaxOffsetY : 0;

    // 몬스터가 오른쪽에서 쐈을 때 화살촉이 왼쪽(플레이어 방향)을 향하도록 반전
    m_bFlipX = (iColDiff > 0);
}
```

좌우 반전 시 `GdiTransparentBlt`가 음수 폭을 지원하지 않아 화살이 아예 그려지지 않는 문제가 있었다. `StretchBlt`는 음수 폭으로 반전이 가능하므로, 임시 DC에 반전 복사 후 `TransparentBlt`로 그리는 방식으로 해결했다.

```cpp
if (m_bFlipX)
{
    HDC hTempDC = CreateCompatibleDC(_DC);
    HBITMAP hTempBmp = CreateCompatibleBitmap(_DC, 21, 21);
    HBITMAP hOldBmp = (HBITMAP)SelectObject(hTempDC, hTempBmp);

    RECT rc = { 0, 0, 21, 21 };
    FillRect(hTempDC, &rc, (HBRUSH)GetStockObject(WHITE_BRUSH));
    StretchBlt(hTempDC, 20, 0, -21, 21, hArrowDC, 0, 0, 21, 21, SRCCOPY);

    GdiTransparentBlt(_DC, iDestX, iDestY, 21, 21,
        hTempDC, 0, 0, 21, 21, RGB(255, 255, 255));

    SelectObject(hTempDC, hOldBmp);
    DeleteObject(hTempBmp);
    DeleteDC(hTempDC);
}
```

## 2. 플레이어/몬스터 좌우 반전

원본 스프라이트가 왼쪽을 향한 한 장뿐이라, 오른쪽으로 이동할 때 반대 방향 스프라이트가 없어 어색했다. `NewPlayer`는 이동 키 입력 시, `NewMonster`는 `Chase()` 내에서 다음 칸의 col 방향에 따라 `m_bFlipX`를 세팅하고, 화살과 동일한 임시 DC 반전 방식으로 `Render()`에서 처리했다. `NewMonster`에 `m_bFlipX`를 두어 `NewMonster::Render()`를 공유하는 `Monster_Melee`, `Monster_Range` 양쪽 모두 한 번의 수정으로 적용됐다.

```cpp
// NewMonster::Chase() 내
if (_next.col > m_tGridPos.col)
    m_bFlipX = true;
else if (_next.col < m_tGridPos.col)
    m_bFlipX = false;
// col이 같으면(상하 이동) 현재 방향 유지
```

## 3. 던전 구성 (4방 + 통로)

기존에는 테두리만 벽인 방 하나뿐이었다. 에디터 씬(`CMyEdit`)이 이미 게임 프로젝트 내에 구현되어 있어 별도 에디터 프로젝트 없이 활용했다. 맵 전체를 한눈에 볼 수 없는 문제는 Python으로 `Tile.dat`을 직접 생성하는 방식으로 해결했다. 저장 포맷(타일 하나당 `float fX, fY, int iCX, iCY, DrawID` = 20바이트)을 파싱해 4방 + 통로 배치를 코드로 생성한 뒤, 에디터에서 세부 수정하는 방식으로 진행했다.

맵 크기는 `TILEX(50) × TILEY(35)`로 확장했다. 모든 관련 시스템(TileMgr, Sight, PathFind, ScrollMgr, NewPlayer)이 `Define.h`의 상수를 참조하는 구조라 상수 변경만으로 전 시스템에 자동 반영됐다.

## 4. 문 타일 추가

방과 방 사이 통로에 시각적 구분이 없었다. `Is_Wall()`이 `DrawID >= WALL_DRAWID(9)`를 벽으로 판정하는 구조를 활용해, 문은 DrawID 2(세로 통로용 정면 문), DrawID 3(가로 통로용 측면 문)으로 지정해 통과 가능하게 했다. 스톤샤드 에셋에서 추출한 문 스프라이트를 26×26으로 크롭해 `Tile.bmp`의 해당 슬롯에 직접 합성했다. 코드 변경 없이 스프라이트 시트 교체만으로 적용 가능한 구조다.

---

## 5. 방향별 벽 타일

기존 벽 스프라이트가 한 방향만 있어 모든 방향의 벽이 같은 텍스처로 보였다. 스톤샤드 에셋(`s_bastion_up_front`, 26×26)을 추출해 Python `PIL.Image.rotate()`로 4방향 회전본을 생성하고, `Tile.bmp`의 DrawID 9~12 슬롯에 각각 합성했다. `Is_Wall()`의 `DrawID >= 9` 판정이 유지되므로 충돌 로직 수정이 필요 없었다.

## 6. BFS 기반 시야 시스템

기존 거리 기반 시야(`iDistance <= PLAYER_SIGHT_RANGE`)는 벽 너머까지 시야가 관통되는 문제가 있었다. 방이 4개로 나뉜 던전 구조에서 다른 방이 보이는 것은 의도하지 않은 동작이었다.

이중 for문 방식을 유지하면서 벽 차폐를 추가하려면, 타일마다 플레이어→타일 방향으로 직선을 그어 중간에 벽이 있는지 검사하는 레이캐스팅이 필요해서 코드가 복잡해진다. BFS는 탐색 자체가 경로를 따라가기 때문에 벽을 만나는 순간 멈추면 자동으로 차폐가 구현된다. 별도 경로 계산 없이 벽 차폐와 거리 제한을 동시에 처리할 수 있어 BFS로 교체했다.

```cpp
void CSight::Update_Sight()
{
    // 기존 VISIBLE → EXPLORED (이전 방은 탐험 상태로 전환)
    for (int i = 0; i < m_iSize; ++i)
        if (m_vecVisibility[i] == VISIBLE)
            m_vecVisibility[i] = EXPLORED;

    vector<bool> visited(m_iSize, false);
    queue<pair<GRIDPOS, int>> q;
    q.push({playerPos, 0});

    while (!q.empty())
    {
        auto [cur, dist] = q.front(); q.pop();
        m_vecVisibility[cur.row * TILEX + cur.col] = VISIBLE;

        if (dist >= PLAYER_SIGHT_RANGE) continue;

        for (int d = 0; d < 8; ++d)
        {
            // 벽이면 EXPLORED로 표시 후 탐색 중단 → 방 경계 역할
            if (CTileMgr::Get_Instance()->Is_Wall(nr, nc))
            {
                m_vecVisibility[idx] = EXPLORED;
                continue;
            }
            q.push({GRIDPOS(nr, nc), dist + 1});
        }
    }
}
```

시야 상태는 단일 벡터(`m_vecVisibility`)로 UNSEEN/VISIBLE/EXPLORED 세 가지를 관리한다. 이동 시 VISIBLE→EXPLORED 1차 전환 후 BFS로 현재 시야를 다시 VISIBLE로 덮어쓰는 방식이라, 별도의 탐험 기록 배열 없이도 세 상태를 일관되게 유지할 수 있다.
