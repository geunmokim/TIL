# 모작 제작 과정 — 몬스터 행동 구현과 던전 맵 제작

## 1. 몬스터 클래스 확장 — Chase 공통화

근접/원거리 몬스터를 만드는 과정에서 "다음 칸으로 한 걸음 이동하되, 그 칸이 점유 중이면 멈춘다"는 동작이 두 클래스에서 완전히 동일하다는 것을 확인했다. 처음 설계에서 공통 부모(NewMonster)에 분류했던 기준("근접이든 원거리든 차이가 없는 동작은 부모에 둔다")을 그대로 적용해, Chase() 함수를 NewMonster로 옮기고 자식 클래스에서는 제거했다. 자식이 이 함수를 호출할 수 있도록 접근 지정자를 private이 아닌 protected로 두었다.

### 핵심 코드 — 공통 이동 로직을 부모 클래스로 이동
```cpp
// NewMonster.h (protected)
void Chase(const GRIDPOS& _next);

// NewMonster.cpp
void CNewMonster::Chase(const GRIDPOS& _next)
{
	if (CObjMgr::Get_Instance()->Is_Occupied(_next))
		return;
	Set_GridPos(_next.row, _next.col);
}
```

## 2. Monster_Melee::On_Turn() 완성

거리 기반으로 공격/추적을 분기하는 로직을 작성했다. 시야범위(MONSTER_SIGHT_RANGE, 임시 상수)를 벗어나면 아무 행동도 하지 않고, 범위 안에서는 거리가 1이면 공격, 그 외에는 PathFind로 다음 칸을 구해 추적하도록 구성했다. 데미지 시스템이 아직 없어 Attack()은 자리만 잡아둔 상태로 비워두었는데, 이는 스탯/데미지 시스템이 인벤토리·장비와 함께 2순위로 분류된 항목이라 1순위 단계에서는 "공격이 발생한다"는 행동 자체만 확인하면 충분하다고 판단했기 때문이다.

### 핵심 코드 — 시야범위와 사거리에 따른 행동 분기
```cpp
void CMonster_Melee::On_Turn()
{
	GRIDPOS playerPos = CObjMgr::Get_Instance()->Get_Player()->Get_GridPos();
	GRIDPOS monsterPos = Get_GridPos();
	int iDistance = monsterPos.Get_Distance(playerPos);

	if (iDistance > MONSTER_SIGHT_RANGE)
		return;

	if (iDistance == 1)
		Attack(CObjMgr::Get_Instance()->Get_Player());
	else
		Chase(CPathFind::Find_NextStep(monsterPos, playerPos));
}
```

## 3. Monster_Range — 사거리를 멤버 변수로 설계

원거리 몬스터의 공격 사거리를 어떻게 표현할지 논의했다. 처음에는 별도 함수로 계산하는 방식을 떠올렸으나, 사거리는 "매번 계산해서 알아내는 값"이 아니라 "몬스터 종류마다 고정되는 값"이라는 점에서, 함수가 아니라 멤버 변수(m_iAttackRange)로 두고 Initialize()에서 설정하는 방식이 적절하다고 결론지었다. 이렇게 하면 나중에 다른 원거리 몬스터 종류를 추가할 때도 클래스를 새로 만들지 않고 같은 클래스에 다른 값만 넣어 재사용할 수 있다.

탄환(m_iAmmo)도 같은 방식으로 멤버 변수로 두었다. 탄환을 충분히 주어 소진될 일이 거의 없게 단순화하기로 했는데, 그 이유로 "사거리 밖으로 도망가며 공격하는" AI를 함께 넣으면 무한 탄환과 결합되어 플레이어가 근접 캐릭터일 경우 영원히 적을 잡을 수 없는 상황이 생길 수 있다는 점을 짚었다. 이에 따라 도망 로직은 넣지 않고, 사거리 안이면 그 자리에서 공격하는 단순한 형태로 유지하기로 했다.

## 4. 화살 투사체(Arrow) 클래스 설계

원거리 공격을 시각적으로 표현하기 위한 화살 클래스를 설계했다. 원작에서 원거리 공격이 매우 빠르게 날아와 명중하는 것을 직접 확인한 뒤, 화살이 여러 프레임에 걸쳐 날아가는 모습을 구현하지 않고 목표 칸에 한 프레임만 나타나고 사라지는 방식으로 단순화하기로 했다.

기존 Obj의 Initialize()가 매개변수를 받지 않는 순수 가상함수라는 제약 때문에, 화살의 목표 위치를 Initialize()에서 직접 받을 수 없다는 점을 확인했다. 위치는 별도의 Set_TargetPos() 함수로 따로 전달하는 방식으로 해결했다.

오브젝트 매니저가 한 프레임에 죽음 처리된 객체를 즉시 리스트에서 제거하는 구조라는 점을 검증 중에 직접 발견했다. 그대로 두면 화살이 생성된 프레임에 그려지지도 못하고 사라지는 문제가 생기므로, 한 프레임의 유예를 두는 카운터(m_iLifeFrame)를 추가해 생성된 다음 프레임에 죽도록 처리했다.

### 핵심 코드 — 한 프레임만 그려지고 사라지도록 하는 유예 처리
```cpp
void CArrow::Initialize()
{
	...
	m_iLifeFrame = 1;
}

int CArrow::Update()
{
	if (m_bDead)
		return OBJ_DEAD;
	if (m_iLifeFrame <= 0)
	{
		m_bDead = true;
		return OBJ_DEAD;
	}
	--m_iLifeFrame;
	return OBJ_NOEVENT;
}
```

화살의 충돌 판정 모양에 대해서도 논의했는데, 화살 스프라이트 자체는 길쭉한 모양이지만 충돌 판정은 원형이 아니라 사각형으로 단순하게 처리하기로 했다. 기존 프레임워크에 이미 사각형과 원형 충돌 판정이 모두 마련되어 있어, 그리드 기반 게임에서는 캐릭터가 위치한 칸 크기만큼의 사각형 판정이 회전 방향을 고려할 필요 없이 더 단순하다고 판단했다.

새로운 오브젝트 분류가 필요해, 기존 OBJID에 ARROW 항목을 추가했다. 이때 enum 중간에 새 항목을 끼워넣으면 그 뒤의 항목들이 가리키는 숫자가 모두 밀린다는 점을 확인했고, 기존 코드가 전부 숫자가 아닌 이름(OBJID::이름)으로 참조하고 있는지 검토해 안전하게 추가할 수 있음을 확인했다.

## 5. 던전 타일 에셋 분석

캐릭터 스프라이트와 달리 던전 타일은 여러 변형이 한 장의 이미지에 가로로 이어붙여진 스트립 형태로 추출되는 경우가 많다는 것을 확인했다.

벽 후보로 검토했던 한 무리의 스프라이트는 크기가 제각각이라는 점에서 "부서진 벽 잔해 장식물"이며 일반 벽이 아니라는 결론에 도달했다. 대신 26x26 크기의 10가지 변형을 가진 벽 타일을 확보했고, 실제 게임 화면과 직접 비교해 색감과 질감이 일치함을 확인하고 채택했다. 바닥 타일은 26x26 크기 9개가 가로로 이어진 234x26 스트립임을 크기 계산으로 확인했다.

## 6. 스트립 분리 및 Tile.bmp 통합

바닥 스트립을 26픽셀 단위로 9등분해 개별 타일로 분리하고, 구분선을 넣은 확대 이미지로 분리 결과를 직접 검증했다. 벽과 바닥 모두 PNG에서 BMP로 변환했으며, 투명 배경은 흰색으로 합성해 기존 투명 처리 코드와 호환되도록 했다.

기존 렌더링 구조가 "한 장의 긴 이미지에서 DrawID만큼 이동해 잘라 그리는" 방식이라는 점에 맞춰, 분리했던 19개 타일을 다시 하나의 이미지(Tile.bmp)로 합쳤다. 배치 순서를 정할 때 기존 WALL_DRAWID 규칙과 일치시키는 과정에서 시행착오를 겪었고, 최종적으로 바닥(DrawID 0~8)을 먼저, 벽(DrawID 9~18)을 나중에 배치하는 것으로 정리하면서 WALL_DRAWID 값도 함께 수정했다.

### 핵심 코드 — 하나의 이미지에서 인덱스로 원하는 타일을 골라 그리는 방식
```cpp
BitBlt(_DC, ...
	, hTileDC
	, m_iDrawID * TILECX, 0
	, SRCCOPY);
```

## 7. 렌더링 크기 불일치 버그 수정

타일을 그리드에 맞춰 그려보니 크기가 맞지 않는 문제를 발견했다. 원인은 두 가지였다. 크기 변환이 불가능한 BitBlt 함수를 쓰고 있었던 것과, 원본에서 잘라올 때의 이동 간격이 실제 원본 타일 크기(26픽셀)가 아니라 그리드 크기(64픽셀) 기준으로 되어 있던 것이다. BitBlt를 StretchBlt로 교체하고 원본 간격 값을 수정해 해결했다.

### 핵심 코드 — 원본 크기를 목적지 크기로 확대해서 그리기
```cpp
StretchBlt(_DC
	, m_tRect.left + iScrollX, m_tRect.top + iScrollY
	, m_tInfo.iCX, m_tInfo.iCY
	, hTileDC
	, m_iDrawID * TILE_SRC_SIZE, 0
	, TILE_SRC_SIZE, TILE_SRC_SIZE
	, SRCCOPY);
```

## 8. 인게임 에디터 확장

던전 배치를 좌표로 코드에 직접 적을지, 기존에 만들어둔 인게임 에디터를 확장해 마우스로 직접 그릴지 검토했다. 추후 다른 그래픽 API로 모작을 다시 만들 계획이 있고, 그런 환경에서는 보통 에디터 툴을 직접 만들어 레벨을 디자인하는 방식이 표준에 가깝다는 점에 착안해 에디터를 확장하는 방향을 선택했다.

기존에 좌클릭/우클릭에 고정된 두 개의 그림 번호만 연결되어 있던 에디터에, 숫자키로 칠할 그림을 선택할 수 있는 기능과 상하 스크롤을 추가했다.

## 9. Picking_Tile() 좌표 보정 버그

맵 가장자리를 클릭했을 때 의도와 다른 위치가 칠해지는 문제를 발견했다. 기존 함수가 가로축 스크롤 보정만 하고 세로축 보정이 빠져 있었던 것이 원인이었다. 스크롤 양이 큰 가장자리일수록 오차가 커지는 구조라, 화면 중앙 근처에서는 거의 드러나지 않고 가장자리에서 두드러지게 나타났던 것으로 분석했다. 세로축 보정을 추가하고, 스크롤이 많이 된 상태와 모서리 케이스를 직접 검증해 해결을 확인했다.
