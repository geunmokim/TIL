# 모작 제작 과정 — BFS 길찾기 & 몬스터 시스템 구현

## 1. PathFind 클래스 — BFS 길찾기

### 왜 필요했나
턴제 시스템에서 몬스터가 플레이어를 추적하려면, 단순히 "플레이어 방향으로 한 칸 이동"만으로는 벽을 만났을 때 막혀버린다. 벽을 우회하면서 항상 최단 경로로 다가오는 길찾기가 필요했다.

### 설계 방향
- 클래스를 싱글톤이 아니라 static 함수로 구성. 길찾기는 "입력(시작점, 목표점) → 계산 → 출력"만 반복하는 순수 계산 작업이라 유지해야 할 상태가 없기 때문. 상태를 가진 매니저(TileMgr, KeyMgr 등)와는 성격이 다르다고 판단했다.
- 탐색에 필요한 방문 기록(visited)과 경로 역추적용 기록(parent)은 멤버 변수가 아니라 함수의 지역 변수로 둠. static 함수에서는 일반 멤버 변수에 접근할 수 없을 뿐 아니라, 매번 호출 시 깨끗한 상태로 시작되어야 하고, 여러 몬스터가 동시에 길찾기를 수행해도 서로 간섭하지 않아야 하기 때문.

### 적용한 방식
- `Find_NextStep(_start, _target)` — 몬스터의 현재 위치(_start)에서 플레이어 위치(_target)까지, 다음 한 턴에 움직여야 할 칸 하나만 반환. 턴제에서는 한 턴에 한 칸만 이동하므로 전체 경로 전부가 아니라 "다음 한 걸음"만 필요하다는 점을 반영했다.
- BFS는 큐(queue)로 칸을 순서대로 탐색하고, 상하좌우(이후 대각선 포함 8방향으로 확장) 방향 배열(dRow, dCol)로 인접 칸을 순회한다.
- 목표에 도달하면, parent 배열을 목표 지점에서부터 거슬러 올라가며 추적하다가 "출발점의 바로 다음 칸"에서 멈춰 그 좌표를 반환한다.
- `_start == _target`(이미 같은 칸)인 경우 parent 배열에 출발점 자신의 부모가 기록될 수 없어 무한루프에 빠지는 문제를 발견해, 함수 맨 앞에서 즉시 반환하도록 처리했다.
- TileMgr::Is_Wall()을 그대로 재사용해 벽 체크를 수행했다.

### 검증
직진 경로, 벽 우회, 동일 위치, 인접 위치, 도달 불가능한 경우 등 다섯 가지 케이스를 직접 컴파일·실행해 모두 정상 동작함을 확인했다.

### 핵심 코드 — 부모 기록을 거슬러 올라가 다음 한 걸음만 추출
```cpp
if (current == _target)   // 목표(플레이어)에 도달했는지 확인
{
    GRIDPOS step = _target;
    while (!(parent[step.row][step.col] == _start))
        step = parent[step.row][step.col];   // 출발점 바로 다음 칸이 나올 때까지 역추적
    return step;   // 전체 경로가 아니라, 이번 턴에 움직일 한 칸만 반환
}
```

## 2. 캐릭터 겹침 방지 — Is_Occupied()

### 왜 필요했나
그리드 이동을 구현하는 과정에서, 플레이어와 몬스터가 같은 칸에 동시에 위치할 수 있는 상태였다는 것을 확인했다. 이동 가능 여부를 판단할 때 "그 칸에 다른 캐릭터가 있는가"를 확인하는 절차가 빠져 있었다.

### 적용한 방식
- ObjMgr에 Is_Occupied(GRIDPOS) 추가. 플레이어와 몬스터 리스트를 순회하며 살아있는 캐릭터가 해당 칸에 있는지 확인한다.
- 죽은 캐릭터는 점유로 취급하지 않도록 처리 — 시체가 있는 칸으로도 다른 캐릭터가 이동할 수 있어야 하기 때문이다.
- 이 작업 중에 "플레이어가 죽으면 턴 진행 자체를 멈춰야 하는가"를 점검했고, 실제로 그런 처리가 빠져 있었다는 것을 발견해 TurnMgr에 게임오버 상태(m_bGameOver)를 추가했다. 몬스터 턴 처리가 끝난 직후 플레이어 생존 여부를 확인해, 몬스터의 공격으로 그 턴 중에 사망한 경우까지 정확히 반영되도록 했다.
- NewPlayer::Can_Move()에 Is_Occupied() 체크를 추가해 몬스터가 있는 칸으로는 이동할 수 없게 했다.

### 핵심 코드 — 살아있는 캐릭터만 점유로 판정
```cpp
bool CObjMgr::Is_Occupied(const GRIDPOS& _pos)
{
    CObj* pPlayer = Get_Player();
    if (pPlayer && !pPlayer->Is_Dead() && pPlayer->Get_GridPos() == _pos)
        return true;

    for (auto& pMonster : m_listObj[OBJID::MONSTER])
    {
        if (pMonster->Is_Dead())
            continue;
        if (pMonster->Get_GridPos() == _pos)
            return true;
    }
    return false;
}
```

## 3. 몬스터 클래스 구조

### 설계 방향
- 기존 액션 게임용 Monster 클래스는 그대로 두고, 그리드 이동에 맞는 새 클래스를 NewPlayer와 같은 방식으로 별도 추가했다.
- 분류 기준으로 "종족(인간/언데드)"과 "전투 방식(근접/원거리)"을 검토했다. 코드 동작에 실질적인 차이를 만드는 기준은 전투 방식 쪽이라고 판단해, 종족 차이는 클래스 분리가 아니라 스프라이트와 스탯 같은 데이터 값으로 처리하기로 했다.
- 공통 부모 클래스(NewMonster)를 만들어 Update, Late_Update, Render, Release처럼 근접/원거리가 동일하게 동작하는 부분을 구현하고, Initialize와 On_Turn처럼 행동이 달라지는 부분은 순수 가상함수로 남겨 자식 클래스(Monster_Melee, Monster_Range)가 각자 구현하도록 했다.
- 원거리 클래스에는 탄환 수(m_iAmmo)를 멤버로 추가했다. 탄환을 충분히 주어 소진될 일이 거의 없게 단순화하기로 했으나, 탄환을 소비한다는 개념 자체는 유지하기로 했다.

### 핵심 코드 — 공통 동작은 구현, 행동이 다른 부분은 자식에게 위임
```cpp
class CNewMonster : public CObj
{
public:
    virtual void Initialize() override = 0;   // 스프라이트가 다르므로 자식이 구현
    virtual int Update() override;             // 공통 구현
    virtual void Render(HDC _DC) override;     // 공통 구현
    virtual void On_Turn() override = 0;       // 근접/원거리 행동이 다르므로 자식이 구현
};
```

## 4. 대각선(8방향) 이동

### 왜 필요했나
원작이 8방향 이동을 지원한다는 점을 확인하고, 4방향만 고려했던 기존 설계를 확장하기로 했다.

### 적용한 방식
- PathFind의 방향 배열(dRow, dCol)에 대각선 방향 4개를 추가하고 순회 범위를 4에서 8로 늘렸다. 나머지 탐색 로직은 방향 개수와 무관하게 동작하도록 짜여 있어 그대로 재사용할 수 있었다.
- 플레이어 입력 처리에서도 대각선 동시 입력을 인식하도록 수정했다. 각 방향키의 눌림 상태를 개별적으로 확인한 뒤 조합해서 판단하는 방식으로 변경했다.
- 대각선 이동 시 인접한 두 칸이 모두 벽이어도 그 모서리를 가로질러 이동하는 현상(코너 커팅)이 발생할 수 있다는 점을 메모로 남겼다. 실제 플레이 테스트에서 부자연스럽게 느껴지면 대각선 이동 전 인접 칸을 추가로 확인하는 처리를 더할 수 있다.

### 핵심 코드 — 방향 배열만 확장, 나머지 탐색 로직은 그대로 재사용
```cpp
// 4방향 -> 8방향으로 확장한 부분
int dRow[] = { -1,1,0,0, -1,-1,1,1 };
int dCol[] = { 0,0,-1,1, -1,1,-1,1 };

for (int i = 0; i < 8; ++i)   // 4 -> 8로 변경, 아래 로직은 손대지 않음
{
    int nextRow = current.row + dRow[i];
    int nextCol = current.col + dCol[i];
    // 범위/벽/방문 체크는 방향 개수와 무관하게 동일
}
```

## 5. 체비셰프 거리 — 인접 판정 기준 변경

### 왜 필요했나
8방향 이동으로 확장하면서, 기존에 4방향만 인접으로 인정하던 거리 계산 방식(맨해튼 거리)이 더 이상 정확하지 않게 되었다. 대각선으로 붙어 있는 경우도 인접으로 인정해야 했다.

### 적용한 방식
- GRIDPOS 구조체에 Get_Distance() 멤버 함수를 추가했다. 두 칸의 행 차이와 열 차이 중 더 큰 값을 거리로 계산하는 체비셰프 거리 방식으로, 8방향 이동에서의 최소 이동 횟수를 정확히 나타낸다.
- 이 거리 계산 함수는 공격 가능 여부(거리가 1인 경우) 판정과, 이후 추가할 발견(시야) 범위 판정에 공통으로 재사용할 수 있도록 설계했다.
- 거리 계산을 PathFind가 아니라 GRIDPOS 자체에 멤버로 둔 이유는, PathFind는 길찾기라는 단일 책임만 갖도록 유지하고 거리 계산은 좌표 데이터 자체의 기능으로 보는 것이 더 자연스럽다고 판단했기 때문이다.

### 핵심 코드 — 체비셰프 거리 계산
```cpp
// row 차이와 col 차이 중 더 큰 값 = 8방향 이동에서의 최소 걸음 수
int Get_Distance(const tagGridPos& _Other) const
{
    int iRowDiff = abs(row - _Other.row);
    int iColDiff = abs(col - _Other.col);
    return max(iRowDiff, iColDiff);
}
```
